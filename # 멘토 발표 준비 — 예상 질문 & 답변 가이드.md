## PART 0 — 먼저 알아둘 핵심 용어 한 줄 정리

| 용어 | 한 줄 설명 |
|------|-----------|
| **RAG** | 외부 문서를 검색해서 LLM에 넘겨주는 방식. 거짓말(할루시네이션) 방지 |
| **LLM** | GPT처럼 글을 이해하고 생성하는 대형 언어 모델 |
| **임베딩** | 텍스트를 숫자 벡터로 변환하는 것. 의미가 비슷하면 숫자도 비슷해짐 |
| **벡터 DB** | 임베딩 숫자를 저장하고 빠르게 유사한 것을 찾아주는 데이터베이스 |
| **청크 (Chunk)** | 긴 문서를 작게 잘라놓은 조각. 이 프로젝트에서는 200토큰 단위 |
| **하이브리드 검색** | 벡터 검색 + 키워드 검색을 섞어서 쓰는 방식 |
| **BM25** | Best Match 25의 약자. 단어 빈도·희귀도·문서 길이로 관련도를 점수화하는 키워드 검색 알고리즘 |
| **TF** | Term Frequency. 단어가 문서 안에 나온 횟수. BM25 구성 요소 |
| **IDF** | Inverse Document Frequency. 전체 문서에서 희귀한 단어일수록 높은 가중치. BM25 구성 요소 |
| **Reranker** | 검색 결과를 한 번 더 정밀하게 재정렬해주는 모델 |
| **Cross-Encoder** | 질문과 문서를 함께 보고 관련도를 판단하는 방식 (Reranker가 이 방식) |
| **Bi-Encoder** | 질문, 문서를 따로 임베딩하고 유사도 비교 (BGE-M3가 이 방식) |
| **Hit@K** | 상위 K개 검색 결과 안에 정답이 있는 비율. 높을수록 좋음 |
| **MRR** | 정답이 몇 번째에 나왔는지의 역수 평균. 높을수록 좋음 |
| **LLM-as-Judge** | GPT가 다른 모델의 답변을 채점하는 평가 방식 |
| **HyDE** | 질문으로 가상 답변을 만들어 그걸 임베딩하는 고급 검색 기법 |
| **쿼리 확장** | 질문 1개를 여러 표현으로 늘려 더 넓게 검색하는 기법 |
| **할루시네이션** | LLM이 사실이 아닌 내용을 자신있게 말하는 현상 |
| **pgvector** | PostgreSQL(관계형 DB)에서 벡터 검색을 가능하게 해주는 확장 |
| **HNSW** | 벡터 검색을 빠르게 해주는 인덱스 알고리즘 (Supabase에서 사용) |
| **MTEB** | 임베딩 모델 성능을 비교하는 글로벌 벤치마크 |
| **Supabase** | PostgreSQL 기반의 무료 클라우드 데이터베이스 서비스 |
| **Streamlit** | Python으로 웹 UI를 빠르게 만들 수 있는 프레임워크 |
| **HuggingFace Spaces** | 머신러닝 앱을 무료로 배포할 수 있는 플랫폼 |
| **Groq** | Llama 모델을 무료로 API로 제공해주는 서비스 |
| **스트리밍** | 답변을 한 번에 주는 게 아니라 생성되는 대로 실시간 출력 |

## PART 1 — RAG 기본 개념 질문

---

### Q1. RAG가 뭔가요? 왜 사용했나요?

**답변:**
> RAG는 Retrieval-Augmented Generation의 약자로, "검색 + 생성"을 합친 방식입니다.
>
> GPT 같은 LLM은 학습된 지식만으로 답변하기 때문에, 최신 정보가 없거나 세부 사실을 모르면 그럴듯하게 거짓말을 할 수 있습니다. 이를 **할루시네이션**이라고 합니다.
>
> RAG는 이를 해결하기 위해 **먼저 관련 문서를 검색한 뒤, 그 내용을 LLM에게 "참고 자료"로 제공하고 답변을 생성**하게 합니다.
>
> 저는 건강식품·다이어트 논문 420개를 DB에 저장해두고, 사용자가 질문하면 그 논문에서 관련 내용을 찾아 GPT에게 넘겨주는 방식으로 구현했습니다.

**핵심 한 줄:** "LLM이 모르는 전문 지식을 외부 문서에서 찾아 주입해서 정확하게 답변하게 하는 방식"

---

### Q2. 임베딩이 뭔가요?

**답변:**
> 텍스트를 컴퓨터가 비교할 수 있는 숫자 벡터로 변환하는 것입니다.
>
> 예를 들어 "오메가3"와 "EPA DHA"는 단어는 다르지만 의미가 비슷하기 때문에, 좋은 임베딩 모델은 이 두 텍스트를 숫자 공간에서 가까운 위치에 놓습니다.
>
> 이 프로젝트에서는 `BAAI/bge-m3` 모델을 사용해서 텍스트를 1024개의 숫자(1024-dim 벡터)로 변환합니다. 질문과 문서를 같은 방식으로 임베딩하면, 숫자 간 거리로 "얼마나 관련 있는지" 계산할 수 있습니다.

**핵심 한 줄:** "텍스트를 숫자로 바꿔서 의미적 유사도를 비교할 수 있게 만드는 것"

---

### Q3. 벡터 DB가 왜 필요한가요? 그냥 DB 쓰면 안 되나요?

**답변:**
> 일반 DB는 정확한 값 일치 검색(LIKE, = 등)만 가능합니다. 하지만 임베딩 검색은 "1024개 숫자 중에서 가장 비슷한 벡터를 빠르게 찾기"가 필요합니다.
>
> 18,682개 청크의 벡터 전체를 매번 비교하면 너무 느리기 때문에, **HNSW**라는 인덱스 알고리즘을 사용해서 빠르게 근사 최근접 이웃을 찾습니다.
>
> 저는 Supabase의 pgvector 확장을 사용했는데, PostgreSQL(관계형 DB) 위에서 벡터 검색을 함께 할 수 있어서 추가 인프라 없이 무료로 운영할 수 있었습니다.

---

### Q4. 청크(Chunk)란 뭔가요? 왜 200토큰으로 잘랐나요?

**답변:**
> 논문 PDF 한 편이 수십 페이지에 달하는데, 전체를 통째로 임베딩하면 너무 길어서 어떤 내용이 들어있는지 뭉개집니다. 그래서 적당한 크기로 잘라서 각 조각을 개별적으로 임베딩합니다.
>
> 처음에는 400토큰으로 시작했다가, 실험을 통해 **200토큰이 검색 정밀도와 내용 포함 범위 사이의 최적점**임을 확인하고 재청킹했습니다.
>
> - 너무 짧으면 (50토큰): 맥락이 없어서 답변이 불완전
> - 너무 길면 (500토큰): 임베딩이 뭉개지고 검색 정확도 하락
> - 200토큰: 논문 한 단락 분량, 충분한 맥락 + 정밀한 검색

## PART 2 — 검색 파이프라인 질문

---

### Q4-1. BM25가 뭔가요? BM은 무슨 뜻인가요?

**답변:**
> BM25는 **Best Match 25**의 약자입니다. "가장 잘 맞는 것을 찾는다"는 의미이고, 25는 개발 과정에서 거쳐온 버전 번호입니다.
>
> 구글 초창기 검색 엔진, Elasticsearch, PostgreSQL FTS 등에서 기본으로 쓰이는 **키워드 기반 검색 알고리즘**입니다.
>
> 점수를 계산할 때 세 가지 요소를 봅니다:
>
> | 요소 | 의미 | 예시 |
> |------|------|------|
> | **TF** (단어 빈도) | 문서 안에 많이 나올수록 ↑ | "오메가3"가 5번 나온 청크가 1번 나온 청크보다 높은 점수 |
> | **IDF** (역문서 빈도) | 전체 문서에서 희귀한 단어일수록 ↑ | "EPA-DHA 이중맹검"처럼 드문 단어가 "건강"처럼 흔한 단어보다 점수 높음 |
> | **문서 길이** | 짧은 문서에서 나온 단어가 ↑ | 200자 청크에서 "오메가3" 등장 > 2000자 청크에서 등장 |
>
> 이 프로젝트에서는 PostgreSQL의 FTS(Full Text Search) 기능이 BM25를 내부적으로 구현하고 있고, 하이브리드 검색에서 **30% 비중**으로 벡터 검색을 보완합니다.
>
> **BM25가 특히 유리한 경우:**
> - 고유명사·브랜드명: "푸드올로지", "비비랩"
> - 수치·단위: "1일 1000mg", "12주 임상"
> - 영어 약어: "RCT", "EPA", "BMI"

**핵심 한 줄:** "단어가 문서에 얼마나 자주, 얼마나 독특하게 등장하는지로 관련도를 점수 매기는 키워드 검색 알고리즘"

---

### Q5. 하이브리드 검색이란 뭔가요?

**답변:**
> 두 가지 검색 방법을 섞어서 쓰는 방식입니다.
>
> **벡터 검색 (70%)**: 의미 기반. "오메가3 심혈관 효능"을 물으면 "EPA DHA heart health"처럼 표현이 달라도 찾을 수 있음.
>
> **BM25 키워드 검색 (30%)**: 단어 일치 기반. "푸드올로지", "특정 브랜드명" 같은 고유명사를 정확히 매칭할 때 유리.
>
> 둘을 섞으면 각각의 단점을 보완할 수 있습니다. 70:30 비율은 실험 5~7을 통해 최적값으로 확인했습니다.

---

### Q6. 왜 가중치를 70:30으로 했나요? 다른 비율은요?

**답변:**
> 실험적으로 검증했습니다. 같은 BGE-M3 임베딩 모델로 세 가지 비율을 비교했을 때:
>
> | 비율 | Hit@5 | MRR |
> |------|-------|-----|
> | 벡터 70 : 키워드 30 | **0.781** | **0.652** |
> | 벡터 60 : 키워드 40 | 0.774 | 0.641 |
> | 벡터 50 : 키워드 50 | 0.761 | 0.629 |
>
> 처음 설정한 70:30이 가장 좋았습니다. 이는 건강식품 논문 특성상 의미 기반 검색이 더 중요하기 때문입니다.

---

### Q7. Reranker가 뭔가요? 왜 필요한가요?

**답변:**
> 처음 검색은 빠른 대신 정밀도가 부족합니다. Reranker는 검색으로 추려진 후보들을 다시 한번 더 정밀하게 평가해 순서를 재배열합니다.
>
> **Bi-Encoder (BGE-M3)**: 질문과 문서를 각각 따로 임베딩 → 빠르지만 둘을 함께 보지 않음
>
> **Cross-Encoder (BGE Reranker)**: 질문과 문서를 쌍으로 같이 넣어서 관련도 판단 → 느리지만 정밀함
>
> 그래서 "빠른 BGE-M3로 Top-5를 뽑고, 정밀한 Reranker로 Top-3를 선별"하는 2단계 구조를 씁니다. 실험 4에서 이를 도입한 후 Hit@3이 25% 향상됐습니다.

---

### Q8. HyDE가 뭔가요?

**답변:**
> Hypothetical Document Embeddings의 약자입니다.
>
> 기존에는 사용자의 질문 자체를 임베딩해서 검색했는데, 문제는 사용자가 "오메가3 효능이 뭐야?"라고 한국어로 물으면, DB에는 "EPA reduces cardiovascular risk"처럼 영어 논문 표현이 있어서 임베딩 공간에서 거리가 멀어집니다.
>
> HyDE는 이를 해결하기 위해 **LLM에게 먼저 가상의 답변을 생성하게 하고, 그 가상 답변을 임베딩해서 검색**합니다. 가상 답변은 논문 스타일 영어로 나오기 때문에 실제 논문 청크와 표현이 더 유사해집니다.
>
> 실험 10에서 GPT 품질이 +0.214 향상됐습니다.

**핵심 한 줄:** "질문 대신 '논문 스타일 가상 답변'을 임베딩해서 영어 논문과 매칭을 높이는 기법"

---

### Q9. 쿼리 확장(Query Expansion)이란 뭔가요?

**답변:**
> 질문 1개로는 표현 다양성에 한계가 있어서, GPT에게 같은 의미의 질문을 3가지 다른 표현으로 만들어달라고 합니다.
>
> 예: "오메가3의 심혈관 효능은?" →
> - 변형1: "Omega-3 cardiovascular effects RCT"
> - 변형2: "오메가-3 EPA DHA 심장 건강"
> - 변형3: "오메가3 임상 연구 혈중지질"
>
> 원본 + 변형 3개 = 총 4개 쿼리로 병렬 검색 후 결과를 합산(중복 제거)합니다. 평균 수집 청크가 5개 → 14.2개로 늘어나고, Reranker가 이 중 최선의 Top-3를 선별합니다.
>
> 실험 11에서 GPT 품질이 +0.357, 전체 최고 개선이었습니다.

**핵심 한 줄:** "같은 질문을 여러 표현으로 늘려 더 넓게 검색하는 방법 — SQL/DB 변경 없이 코드로만 구현"

## PART 3 — 평가 방법 질문

---

### Q10. Hit@K가 뭔가요?

**답변:**
> "상위 K개 검색 결과 안에 정답 문서가 있는 비율"입니다.
>
> 예를 들어 Hit@5 = 0.781이면, 100번 검색했을 때 78번은 상위 5개 안에 정답 청크가 포함됐다는 의미입니다.
>
> 저는 각 질문마다 "이 질문의 정답이 어느 청크에 있는지(chunk_id)"를 미리 알고 있는 QA 데이터셋을 만들어서, 검색 결과에 그 chunk_id가 포함됐는지 확인했습니다.

---

### Q11. MRR이 뭔가요?

**답변:**
> Mean Reciprocal Rank의 약자로, "정답이 몇 번째에 나왔는지의 역수 평균"입니다.
>
> - 1번째에 나오면: 1/1 = 1.0
> - 2번째에 나오면: 1/2 = 0.5
> - 5번째에 나오면: 1/5 = 0.2
> - 없으면: 0
>
> Hit@K는 "있다/없다"만 측정하지만, MRR은 "얼마나 앞에 나왔는지"까지 반영합니다. 그래서 두 지표를 함께 보는 게 더 완전한 평가입니다.

---

### Q12. LLM-as-Judge가 뭔가요?

**답변:**
> 챗봇이 생성한 답변의 품질을 GPT-4o-mini가 자동으로 채점하는 방식입니다.
>
> 사람이 직접 채점하면 정확하지만 시간이 너무 많이 걸립니다. 그래서 "정답 기준 청크 내용"과 "챗봇이 생성한 답변"을 GPT에게 주고 0~3점으로 채점하게 합니다.
>
> - 0점: 완전히 틀림
> - 1점: 일부 포함
> - 2점: 대부분 정확
> - 3점: 완전 정확
>
> 한계: 채점자 자체가 LLM이라 편향이 있을 수 있습니다. 특히 같은 회사 모델(GPT가 GPT를 채점)이면 유리할 수 있어서, 완벽한 방법은 아닙니다.

---

### Q13. 내부 QA 데이터셋은 어떻게 만들었나요?

**답변:**
> 실제 DB에 있는 청크 42개를 샘플링해서, 각 청크를 읽고 GPT에게 "이 내용에서 질문을 만들어줘"라고 자동 생성했습니다.
>
> 이렇게 하면 각 질문마다 정답 청크(chunk_id)를 정확히 알고 있으므로, 검색 평가(Hit@K)를 할 수 있습니다.
>
> 반면 조원 QA는 직접 손으로 만든 62문항으로, chunk_id가 없어서 답변 품질만 평가했습니다.

## PART 8 — 청킹(Chunking) 개념 완전 정리

> 멘토님이 주요 튜닝 요소로 언급하신 내용. 면접에서 설명할 수 있어야 합니다.

---

### 청킹이란?

긴 문서(논문, PDF)를 LLM과 검색 시스템이 처리하기 좋은 **작은 조각(청크)으로 나누는 작업**입니다.  
청크가 너무 크면 임베딩이 뭉개지고, 너무 작으면 맥락이 사라집니다.

---

### 청킹 방식 4가지 비교

#### 1. Fixed Length (고정 길이 청킹) — ✅ 이 프로젝트에서 사용한 방식

```
[논문 전체 텍스트]
→ 200토큰씩 잘라서 조각 생성
→ 보통 overlap(겹침)을 20~50토큰 두어 경계에서 문장이 잘리는 문제 완화
```

- **장점**: 구현이 단순하고 청크 수 예측 가능
- **단점**: 문장/문단 중간에서 잘릴 수 있음 → 맥락 손실 가능
- **이 프로젝트**: 200토큰 고정, `parse_and_upload_v2.py`에서 구현

#### 2. Recursive Character Splitting (재귀적 분리)

```
[논문 전체]
→ 먼저 "\n\n" (문단 경계)로 시도
→ 너무 길면 "\n" (줄바꿈)으로 시도
→ 그래도 길면 ". " (문장 끝)으로 시도
→ 최후에는 글자 단위로 자름
```

- **장점**: 자연스러운 경계를 최대한 존중 → Fixed보다 맥락 보존 우수
- **단점**: Fixed보다 구현 복잡, LangChain의 `RecursiveCharacterTextSplitter`가 대표적
- **언제 쓰나**: 구조화가 약한 일반 텍스트 문서에 적합

#### 3. Semantic Chunking (의미 기반 청킹)

```
[논문 전체]
→ 각 문장을 임베딩
→ 연속된 문장 간 임베딩 유사도 계산
→ 유사도가 급격히 떨어지는 지점 = 주제 전환 → 거기서 분리
```

- **장점**: 의미적으로 관련된 내용이 한 청크에 모임 → 검색 품질 이론상 최고
- **단점**: 임베딩을 미리 다 해야 해서 시간·비용이 많이 듦, 청크 크기가 일정하지 않음
- **언제 쓰나**: 고품질이 필요하고 전처리 시간이 충분할 때

#### 4. Document-based Chunking (구조 기반 청킹) — 멘토님 권장

```
[논문 PDF]
→ 제목, Abstract, Introduction, Method, Result, Conclusion 구조 인식
→ 각 섹션을 단위로 청킹
→ 섹션 제목을 메타데이터로 저장
```

- **장점**: 논문/보고서처럼 명확한 구조가 있는 문서에서 가장 정확한 청킹
- **단점**: 구조 파싱이 어렵고 PDF마다 형식이 달라 구현 난이도 높음
- **언제 쓰나**: PDF 논문처럼 섹션 구조가 명확한 문서에 이상적

---

### 청킹 방식 한눈에 비교

| 방식 | 구현 난이도 | 맥락 보존 | 검색 품질 | 이 프로젝트 |
|------|:---------:|:-------:|:-------:|:---------:|
| Fixed Length | ⭐ 쉬움 | 보통 | 보통 | ✅ 사용 |
| Recursive | ⭐⭐ 보통 | 좋음 | 좋음 | - |
| Semantic | ⭐⭐⭐ 어려움 | 매우 좋음 | 매우 좋음 | - |
| Document-based | ⭐⭐⭐⭐ 매우 어려움 | 최고 | 최고 | - |

---

### Q. 왜 Document-based가 아닌 Fixed Length를 썼나요?

> 실용적인 이유 두 가지입니다.
>
> 1. **PDF 파서의 한계**: 논문 420개가 PDF 형태인데, 출판사마다 포맷이 달라서 섹션 구조를 자동 인식하기가 어렵습니다.
> 2. **실험 결과로 검증**: 처음 400토큰 → 200토큰으로 바꿨을 때 검색 성능이 향상됐고, 현재 200토큰 Fixed가 이 프로젝트 규모에서 충분히 효과적으로 작동하고 있습니다.
>
> 향후 개선한다면 Document-based 또는 Semantic 방식 도입이 가장 효과적인 개선 포인트가 될 것입니다.

---

### 청크 사이즈와 성능의 관계

```
토큰 수    특징                        적합한 상황
─────────────────────────────────────────────────
50~100    매우 정밀, 맥락 부족          키워드 검색 위주
100~200   정밀도와 맥락의 균형 ← 이 프로젝트  논문, 전문 자료
200~400   맥락 풍부, 임베딩 뭉개짐 시작   긴 설명이 필요한 문서
400~600   임베딩 품질 저하              비추천
```

## PART 9 — LangChain · 하네스 · 에이전트, 이 프로젝트에 어떻게 썼나?

---

### LangChain — 사용 안 했습니다 (직접 구현)

**LangChain이란?**
> RAG 파이프라인, 에이전트, 메모리 등을 모듈로 묶어서 쉽게 조립할 수 있게 해주는 Python 프레임워크입니다. `TextSplitter`, `VectorStore`, `Chain`, `Agent` 등의 컴포넌트를 제공합니다.

**이 프로젝트에서?**
> LangChain을 쓰지 않고 **동일한 기능을 직접 Python 코드로 구현**했습니다.

| LangChain 컴포넌트 | 이 프로젝트의 직접 구현 |
|-------------------|----------------------|
| `RecursiveCharacterTextSplitter` | `parse_and_upload_v2.py`의 200토큰 청킹 |
| `SupabaseVectorStore` | `supabase.rpc("hybrid_search_v2", ...)` 직접 호출 |
| `RetrievalQA Chain` | `hybrid_search → rerank → build_context → GPT` 파이프라인 |
| `ConversationBufferMemory` | `session_state["messages"]` + `user_memory.json` |
| `MultiQueryRetriever` | `generate_queries + multi_query_search` 직접 구현 |
| `HypotheticalDocumentEmbedder` | `generate_hypothetical_doc` 직접 구현 |

**멘토님께 어떻게 설명하나?**
> "LangChain의 추상화 레이어 없이 직접 구현했습니다. 덕분에 각 단계의 동작 방식을 더 깊이 이해할 수 있었고, 실험별로 세밀하게 파이프라인을 변경하기도 편했습니다. 실제 서비스라면 LangChain을 도입해 유지보수성을 높이는 게 맞지만, 학습 목적으로는 직접 구현이 더 가치 있다고 판단했습니다."

---

### 하네스(Evaluation Harness) — 사용 안 했습니다 (자체 구현)

**평가 하네스란?**
> RAG 시스템의 품질을 자동으로 측정해주는 평가 프레임워크입니다. 대표적으로:
> - **RAGAS**: Faithfulness, Answer Relevancy, Context Recall 등을 자동 측정
> - **LangSmith**: LangChain 기반 LLM 앱 추적·평가 플랫폼
> - **EleutherAI LM Eval Harness**: 언어 모델 벤치마크 평가 도구

**이 프로젝트에서?**
> RAGAS나 LangSmith 같은 기존 하네스를 사용하지 않고, **평가 스크립트를 직접 작성**했습니다.

| 역할 | 이 프로젝트의 구현 |
|------|-----------------|
| 검색 품질 측정 | `calc_retrieval_metrics()` → Hit@1/3/5, MRR 직접 계산 |
| 답변 품질 측정 | `judge_answer()` → GPT-4o-mini가 0~3점 채점 (LLM-as-Judge) |
| 실험 자동화 | `evaluate_exp9~11.py` — 42문항 자동 순회·저장 |
| 결과 저장 | JSON + Markdown 자동 생성 |
| 시각화 | matplotlib으로 차트 자동 생성 |

**멘토님께 어떻게 설명하나?**
> "RAGAS 같은 프레임워크 대신 자체 평가 파이프라인을 구축했습니다. chunk_id 기반 정확 매칭으로 Hit@K/MRR을 측정하고, LLM-as-Judge로 답변 품질을 0~3점으로 정량화했습니다. 11회 실험을 모두 같은 기준으로 비교할 수 있어 실험 간 공정한 비교가 가능했습니다."

---

### 에이전트(Agent) — 형식적 프레임워크는 미사용, 개념적으로는 적용

**LLM 에이전트란?**
> LLM이 단순히 답변만 하는 게 아니라, **도구(Tool)를 선택·실행하고 그 결과를 보고 다음 행동을 결정**하는 방식입니다. ReAct, Function Calling, Tool Use 등이 여기 해당합니다.

**이 프로젝트에서 GPT가 맡은 역할들 (에이전트적 활용):**

| GPT 역할 | 코드 위치 | 에이전트 개념 |
|---------|----------|------------|
| 쿼리 확장기 | `generate_queries()` | 입력을 분석해 3개 쿼리 생성 |
| 가상문서 생성기 | `generate_hypothetical_doc()` | HyDE용 가상 답변 생성 |
| 답변 생성기 | `stream_answer()` | 검색 결과 기반 최종 답변 |
| 메모리 추출기 | `extract_memory()` | 대화에서 건강 정보 자동 추출 |
| 평가 판사 | `judge_answer()` | 답변 품질 0~3점 자동 채점 |

> GPT가 "쿼리 생성 → 검색 트리거 → 메모리 추출"을 담당한다는 점에서 에이전트적 구조이지만, LangChain Agent나 OpenAI Function Calling 같은 공식 에이전트 프레임워크를 사용한 건 아닙니다.

**멘토님께 어떻게 설명하나?**
> "공식 에이전트 프레임워크(LangChain Agent, AutoGPT 등)는 사용하지 않았지만, GPT를 단일 역할이 아닌 다중 역할로 활용했습니다. 쿼리 확장, HyDE 가상문서 생성, 메모리 추출, 답변 채점 등 각 단계마다 GPT를 도구처럼 호출하는 구조로, 에이전트의 개념을 직접 구현했다고 볼 수 있습니다."
