## 1. 왜 QA 쌍 평가가 필요한가

### 현재 평가 방식의 한계 (키워드 매칭)

```
질문: "오메가3의 심혈관 건강 효능은?"
키워드: ["오메가3", "EPA", "DHA", "심혈관"]
판정: TOP-K 문서 중 하나라도 키워드 포함 → 성공
```

**문제점:**
- 키워드가 있어도 실제로 질문에 답하지 못하는 문서일 수 있음
- "관련 있음"의 정의가 모호 (키워드 1개라도 있으면 성공?)
- 정확히 어떤 문서가 정답인지 모름 → 순위 평가가 부정확

### QA 쌍 방식의 장점

```
질문: "오메가3가 심혈관에 미치는 구체적인 효과는?"
정답 청크 ID: chunk_id = 4821  (이 청크에서 질문이 만들어짐)
판정: TOP-K 결과에 chunk_id 4821이 포함되면 성공
```

- **정확한 정답**이 있음 (특정 청크 ID)
- Hit Rate / MRR 계산이 더 신뢰도 높음
- 멘토 언급: "DB 내 청크 단위로 랜덤 샘플링 → 질문 생성 → 정답 청크 매핑"

---

## 2. 평가 방법론

### 2.1 멘토 언급 내용 정리

> "DB 내 청크 단위로 랜덤 샘플링, 각 청크 기반으로 질문 생성, 정답 청크 매핑 → 질문-정답 쌍 데이터셋 구축"

이것은 **LLM-generated QA pairs** 방식입니다.

### 2.2 전체 흐름

```
[Supabase documents_v2]
        ↓ 랜덤 샘플링 (카테고리별 균등)
[청크 N개 선택]
        ↓ LLM에게 "이 청크를 보고 질문을 만들어줘"
[질문 생성 (GPT-4o-mini)]
        ↓ (질문, 정답_청크_ID) 쌍 저장
[QA 데이터셋 JSON]
        ↓ 각 질문으로 RAG 검색 실행
[검색 결과 Top-K]
        ↓ 정답 청크 ID가 포함됐는지 확인
[Hit Rate @1/3/5, MRR 계산]
        ↓
[결과 마크다운 저장]
```

### 2.3 현재 방식과 비교

| 항목 | 키워드 매칭 (실험 1~7) | QA 쌍 방식 (이번) |
|------|----------------------|-----------------|
| 정답 정의 | 키워드 포함 여부 | 특정 청크 ID |
| 정확도 | 보통 | 높음 |
| LLM 사용 | 없음 | 질문 생성 시 1회 |
| 반복 실험 적합성 | 높음 | 중간 (데이터셋 고정 후 높음) |
| 멘토 권장 | 언급 | **직접 언급** |

## 3. 데이터셋 생성 계획

### 3.1 샘플링 전략

총 **50개** QA 쌍 생성 (카테고리별 균등 배분)

| 카테고리 | 청크 수 (DB) | 샘플 수 |
|---------|------------|--------|
| 다이어트_논문 | ~7,400 | 12 |
| 건강식품_논문 | ~5,500 | 10 |
| 다이어트영양소_논문 | ~1,100 | 8 |
| 건강기사 | ~1,800 | 7 |
| 푸드올로지 | ~162 | 5 |
| 기타 | ~1,200 | 8 |
| **합계** | **18,682** | **50** |

### 3.2 질문 생성 프롬프트

```python
PROMPT = """아래 텍스트를 읽고, 이 텍스트가 직접 답할 수 있는 질문 1개를 만들어줘.

규칙:
- 텍스트에 명시적으로 답이 있는 질문만 만들 것
- 한국어로 작성
- 질문만 출력 (설명 없이)
- 너무 구체적인 숫자/수치 질문은 피할 것

텍스트:
{chunk_content}

질문:"""
```

### 3.3 출력 형식 (qa_dataset.json)

```json
[
  {
    "id": 1,
    "question": "오메가3 EPA와 DHA가 심혈관 질환에 미치는 효과는?",
    "answer_chunk_id": 4821,
    "answer_chunk_content": "오메가3 지방산 중 EPA와 DHA는...",
    "category": "건강식품_논문",
    "source_file": "omega3_EPA_DHA_exercise_physiology_2025.pdf"
  },
  ...
]
```

## 4. 평가 실행 계획

### 4.1 평가 스크립트: `src/evaluate_qa.py`

```
입력: qa_dataset.json (50개 QA 쌍)
처리:
  각 질문 → hybrid_search_v2 검색 → Top-K 결과
  정답 chunk_id가 Top-K 안에 있는지 확인
출력:
  Hit Rate @1 / @3 / @5
  MRR
  실패 케이스 분석
  results/qa_eval_결과.md
```

### 4.2 비교 실험 설정

QA 쌍 평가도 동일하게 실험 조건별로 비교:

| 실험 | 검색 대상 | 임베딩 |
|------|---------|--------|
| QA-E4 | documents (기존) | e5-small |
| QA-E5 | documents_v2 | bge-m3 |

→ 실험 4 vs 실험 5의 실질적 품질 차이를 정답 기반으로 검증

## 5. 실행 순서

```
Step 1. src/generate_qa_dataset.py 작성 및 실행
        - Supabase에서 카테고리별 청크 샘플링
        - GPT-4o-mini로 질문 생성
        - data/generated/qa_dataset.json 저장

Step 2. src/evaluate_qa.py 작성 및 실행
        - QA 쌍 기반 Hit Rate / MRR 계산
        - 실험 4(e5-small) vs 실험 5(bge-m3) 비교
        - results/QA쌍_평가결과.md 저장

Step 3. 결과 분석
        - 키워드 매칭 결과와 QA 쌍 결과 비교
        - 어떤 카테고리에서 검색이 약한지 파악
        - 개선 방향 도출

Step 4. 커밋 및 문서화
```

**"QA 평가를 어떻게 했나요?"**
