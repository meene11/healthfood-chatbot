## 1. 프로젝트 개요

### 목표

건강식품 및 다이어트 분야의 논문·기사·제품 데이터를 기반으로, 사용자 질문에 정확하게 답변하는 **RAG(Retrieval-Augmented Generation) 챗봇** 구축.

### 기술 스택

| 구성 요소 | 기술 |
|---------|------|
| 언어 | Python 3.11 |
| 벡터 DB | Supabase (pgvector 확장) |
| 임베딩 모델 (초기) | `intfloat/multilingual-e5-small` (384차원) |
| 임베딩 모델 (최종) | `BAAI/bge-m3` (1024차원) |
| 리랭커 | `BAAI/bge-reranker-v2-m3` (크로스인코더) |
| LLM | OpenAI GPT-4o-mini |
| RAG 프레임워크 | LangChain |
| 키워드 검색 | BM25 (Supabase RPC 내장) |

## 2. 시스템 아키텍처

### RAG 파이프라인 (최종 버전 — `chatbot_v2.py`)

```
사용자 질문
    │
    ▼
[1] Query Rewrite + Multi-Query
    GPT-4o-mini로 원본 질문을 포함해 검색 쿼리 3개 생성 (API 1회 호출)
    예: "오메가3 효능?" → ["오메가3 심혈관 효능", "EPA DHA 효과", "fish oil 연구"]
    │
    ▼
[2] Hybrid Search (Supabase RPC: hybrid_search_v2)
    벡터 검색 (0.7) + BM25 키워드 검색 (0.3) 가중 합산
    TOP_K=10 문서 반환
    │
    ▼
[3] BGE Cross-Encoder Reranking
    BAAI/bge-reranker-v2-m3로 (질문, 문서) 쌍 관련성 점수 계산
    TOP_K=10 → RERANK_TOP_K=3으로 압축
    │
    ▼
[4] LLM 답변 생성 (GPT-4o-mini, 스트리밍)
    최종 선택된 3개 문서를 컨텍스트로 첨부해 답변 생성
    │
    ▼
[5] Memory Bank (세션 내 사용자 정보 유지)
    사용자 목표, 알레르기, 복용 중인 보조제 등 저장
    후속 질문에 자동 반영
    │
    ▼
사용자에게 답변 반환 (스트리밍)
```

### 데이터 파이프라인

```
[원본 데이터: data/raw/]
  papers/     — PDF 논문 65개 (영어/한국어)
  *.txt       — 한국어 건강기사 6개
  *.json      — 크롤링 데이터
  foodology/  — 푸드올로지 제품 설명
  blog/       — 블로그 포스트
        │
        ▼
[parse_all_data.py]
  PDF: pdfplumber로 페이지별 텍스트 추출
  청킹 전략: Fixed-Length
  초기: CHUNK_TOKENS=400, OVERLAP=60
  최종: CHUNK_TOKENS=200, OVERLAP=40
        │
        ▼
[upload_to_supabase_v2.py]
  임베딩 모델: BAAI/bge-m3 (1024차원)
  배치 처리 후 Supabase documents_v2 테이블 INSERT
        │
        ▼
[Supabase pgvector]
  documents    — 8,760청크, vector(384)  ← 실험 1~4 기준
  documents_v2 — 18,682청크, vector(1024) ← 실험 5~7 기준 (최종)
```

## 4. 실험 결과 (전체 7개 실험)

### 4.1 키워드 매칭 기반 평가 (실험 1~7, 판정 질문 17개)

**평가 방법**: 각 질문에 사전 정의된 핵심 키워드 목록을 작성하고, 검색된 문서에 해당 키워드가 하나라도 포함되면 "관련 있음"으로 판정. Hit Rate(@K)와 MRR(Mean Reciprocal Rank)을 계산.

| 실험 | 변경 항목 | Hit@1 | Hit@3 | Hit@5 | MRR | 주요 발견 |
|------|---------|------:|------:|------:|----:|---------|
| 실험 1: Baseline | 없음 (e5-small, vector 0.7/BM25 0.3, TOP_K=5) | 0.471 | 0.647 | 0.706 | 0.564 | 기준선 |
| 실험 2: 가중치 50/50 | BM25 비중 증가 (e5-small) | 0.471 | 0.647 | 0.706 | 0.564 | 가중치 변경 효과 없음 |
| 실험 3: TOP_K=10 | 검색 범위 확대 | 0.471 | 0.647 | 0.706 | 0.578 | MRR 소폭 향상만 |
| 실험 4: BGE 리랭커 | 크로스인코더 리랭커 교체 | 0.588 | 0.647 | 0.706 | 0.629 | **Hit@1 +11.7%** |
| **실험 5: BGE-M3** | **임베딩 교체 + 재청킹** | **0.765** | **0.882** | **0.941** | **0.828** | **가장 큰 개선** |
| 실험 6: 가중치 60/40 | BM25 비중 증가 (bge-m3) | 0.765 | 0.882 | 0.941 | 0.828 | 변화 없음 |
| 실험 7: 가중치 50/50 | BM25 비중 증가 (bge-m3) | 0.765 | 0.882 | 0.941 | 0.828 | 변화 없음 |

### 4.2 핵심 발견

- **임베딩 모델 교체가 가장 큰 개선 요인**: Hit@5 +23.5%p (0.706 → 0.941), MRR +19.9%p (0.629 → 0.828)
- **가중치 변경은 효과 없음**: 실험 2, 6, 7 모두 동일 — bge-m3 이후 벡터 검색이 이미 충분히 정확해 BM25 비중을 높여도 개선 없음
- **BGE 리랭커는 Hit@1 개선에 효과적**: Hit@1 +11.7%p (0.471 → 0.588), 상위 문서 순위 정밀도 향상
- **평균 유사도 점수는 낮아짐**: 청크 수가 2.1배 늘어나 점수 분산 증가 (0.637 → 0.420). 유사도 점수 자체보다 순위 지표(Hit/MRR)가 더 신뢰성 있는 평가 기준임을 시사

### 4.3 QA 쌍 기반 평가 (42개 질문, 정답 chunk_id 기반)

**평가 방법**: documents_v2에서 카테고리별 균등 샘플링한 42개 청크에 대해 GPT-4o-mini로 질문 생성. (질문, 정답_chunk_id) 쌍 데이터셋으로 RAG 검색을 실행하고 정답 청크가 Top-K에 포함되는지 평가.

| 실험 | Hit@1 | Hit@3 | Hit@5 | MRR |
|------|------:|------:|------:|----:|
| QA-E5: bge-m3 (documents_v2) | 0.024 | 0.071 | 0.095 | 0.048 |

**카테고리별 Hit@5**

| 카테고리 | Hit@5 | 샘플 수 |
|---------|------:|-------:|
| 건강기사 | 0.143 | 7 |
| 건강식품_논문 | 0.000 | 10 |
| 기타 | 0.125 | 8 |
| 다이어트_논문 | 0.167 | 12 |
| 푸드올로지 | 0.000 | 5 |

**낮은 점수의 원인**: Hit@5 = 0.095는 RAG 시스템이 검색을 못 한다는 의미가 아니라, 18,682개 청크 중 특정 청크 하나를 정확히 맞추기 어렵다는 의미.

| 원인 | 설명 |
|------|------|
| 중복 유사 청크 | 동일 논문이 수백 개 청크로 분할됨. 같은 질문에 답할 수 있는 청크가 여러 개 존재 |
| 문맥 의존 질문 | GPT-4o-mini가 일부 질문에 "이 연구", "이 그룹" 같은 문맥 참조 표현 사용 (42개 중 4개) |
| 논문 내부 코드 참조 | "S61은 어떤 사람들입니까?", "RoB 평가가 어떻게 되었습니까?" 등 논문 특정 내부 구조 참조 |
| 엄격한 판정 기준 | 정확히 같은 chunk_id가 있어야 성공. 의미적으로 동일한 내용의 다른 청크는 실패 처리 |

**두 평가 방식 비교**

| 평가 방식 | Hit@5 | MRR | 질문 수 | 특징 |
|---------|------:|----:|-------:|------|
| 키워드 매칭 (실험 5) | 0.941 | 0.828 | 17개 | 사용자 관점 질문, 키워드 존재 여부 판정 |
| QA 쌍 (이번 평가) | 0.095 | 0.048 | 42개 | 논문 청크 기반 질문, 정확한 chunk_id 일치 판정 |

키워드 매칭 평가가 실제 사용자 경험에 더 가까운 지표임.

## 5. 실험별 상세 분석

### 실험 1: Baseline

| 항목 | 설정 |
|------|------|
| 임베딩 | `intfloat/multilingual-e5-small` (384차원) |
| 검색 테이블 | `documents` (8,760청크, 400토큰) |
| 가중치 | vector 0.7 / BM25 0.3 |
| TOP_K | 5 |
| 리랭커 | 키워드 보너스 방식 (크로스인코더 미사용) |

**결과**: Hit@5 = 0.706, MRR = 0.564

**발견 사항**: 17개 질문 중 5개가 꾸준히 실패. 실패 질문 대부분은 한/영 혼재 학술 용어(EPA, DHA, probiotics)를 포함하며, e5-small이 이 의미를 벡터로 잘 표현하지 못하는 것으로 추정.

---

### 실험 2: 하이브리드 가중치 50/50

**변경 사항**: vector_weight 0.7 → 0.5, text_weight 0.3 → 0.5

**의도**: 한국어 질문에서 BM25 키워드 검색 비중을 높이면 정확도가 오를 것으로 기대.

**결과**: Hit@1/3/5 = 실험 1과 동일. MRR = 0.564 (동일). **평균 유사도 점수 하락** (0.637 → 0.455).

**결론**: BM25 비중을 높였더니 오히려 combined_score의 분포가 흐트러져 유사도가 낮아짐. 기존 0.7/0.3이 더 적절한 설정.

---

### 실험 3: TOP_K=10 (검색 범위 확대)

**변경 사항**: TOP_K 5 → 10

**의도**: 관련 문서가 6~10위권에 있다면 범위를 넓혀서 Hit@5 계산 범위 안으로 끌어들일 수 있는지 확인.

**결과**: Hit@1/3/5 = 실험 1과 동일. MRR = 0.578 (소폭 향상 +0.014).

**결론**: TOP_K를 늘려도 관련 문서 자체가 Top5 안에 진입하지 못하는 근본 원인은 임베딩 품질 문제임을 시사. MRR 소폭 향상은 일부 문서의 순위가 개선된 영향.

---

### 실험 4: BGE 크로스인코더 리랭커 교체

**변경 사항**: 키워드 보너스 방식 리랭커 → `BAAI/bge-reranker-v2-m3` 크로스인코더

**배경**: `chatbot_v2.py`에는 `CrossEncoder("cross-encoder/ms-marco-MiniLM-L-6-v2")`가 임포트되어 있었으나 실제 `rerank_documents()` 함수에서는 키워드 보너스 방식(`rerank_score = search_score + keyword_bonus`)만 사용하고 있었음. 즉 크로스인코더가 로드만 되고 실제 사용되지 않는 상태였음.

**의도**: 한국어/영어 모두 지원하는 BGE 크로스인코더로 교체해 Hit@1과 MRR 개선.

> **크로스인코더란?** 일반 임베딩은 질문과 문서를 각각 독립적으로 벡터로 변환 후 코사인 유사도를 계산합니다. 크로스인코더는 (질문, 문서) 쌍을 함께 입력받아 관련성 점수를 직접 계산합니다. 더 정확하지만 문서 수만큼 모델을 반복 실행해야 하므로 느립니다.

**결과**: Hit@1 = 0.588 (+11.7%p), MRR = 0.629 (+6.5%p). Hit@3/5는 변화 없음.

**결론**: 크로스인코더 리랭커는 이미 검색된 문서의 순위를 정밀하게 조정하는 데 효과적. 하지만 애초에 검색 범위(Top5)에 없는 문서는 구제할 수 없어 Hit@5 개선 없음. 이 결과를 `chatbot_v2.py`에 반영하고 main에 병합.

---

### 실험 5: BGE-M3 임베딩 교체 + 재청킹 200토큰 (핵심 실험)

**변경 사항**: 임베딩 모델 + 청크 크기 동시 변경 (두 변수를 함께 바꾼 이유: 재임베딩을 하려면 재청킹도 해야 하므로, 재청킹 비용이 발생하는 시점에 최적 청크 크기도 함께 적용)

| 항목 | 변경 전 | 변경 후 |
|------|---------|---------|
| 임베딩 모델 | e5-small (118M 파라미터, 384차원) | bge-m3 (570M 파라미터, 1024차원) |
| 청크 크기 | 400 토큰 | 200 토큰 |
| 오버랩 | 60 토큰 | 40 토큰 |
| 총 청크 수 | 8,760개 | 18,682개 |
| Supabase 테이블 | `documents` | `documents_v2` (새 테이블, 롤백 가능) |

**결과**: Hit@1 = 0.765 (+17.7%p), Hit@3 = 0.882 (+23.5%p), Hit@5 = 0.941 (+23.5%p), MRR = 0.828 (+19.9%p)

**결론**: 임베딩 모델 교체가 전체 실험에서 가장 큰 성능 개선 요인. bge-m3는 한/영 혼재 학술 텍스트(EPA, DHA, probiotics 등 영어 논문 키워드 + 한국어 질문)에서 e5-small 대비 월등히 좋은 의미 표현 능력을 보임.

---

### 실험 6: BGE-M3 + 가중치 60/40

**변경 사항**: vector 0.7/BM25 0.3 → vector 0.6/BM25 0.4 (bge-m3 기반)

**결과**: 실험 5와 동일. Hit@1/3/5 = 0.765/0.882/0.941, MRR = 0.828.

**결론**: bge-m3 임베딩이 이미 충분히 정확한 벡터를 생성하므로 BM25 비중 증가가 추가 효과를 내지 못함.

---

### 실험 7: BGE-M3 + 가중치 50/50

**변경 사항**: vector 0.7/BM25 0.3 → vector 0.5/BM25 0.5 (bge-m3 기반)

**결과**: 실험 5, 6과 동일. 변화 없음.

**결론**: 가중치 조정은 임베딩 모델만큼의 영향력을 갖지 않음. 기본 설정(0.7/0.3)이 최적.

## 7. 멘토 질문 대비 답변

### Q1. "유사도 평가를 어떻게 했냐? LLM으로 한 거냐?"

**답변:**

LLM을 사용하지 않았습니다.

각 질문마다 사전에 정의한 핵심 키워드 목록을 만들고, 검색 결과 문서에 그 키워드가 포함되어 있으면 "관련 있음"으로 판정했습니다. 이를 기반으로 Hit Rate(@K)와 MRR을 계산했습니다.

LLM 기반 평가(LLM-as-Judge)는 비용과 속도 문제로 제외했습니다. 실험을 7회 반복해야 하는 상황에서 빠르고 재현 가능한 키워드 매칭 방식이 더 적합했습니다.

| 항목 | LLM-as-Judge | 키워드 매칭 (이 프로젝트) |
|------|:-----------:|:----------------------:|
| 정확도 | 높음 | 보통 |
| 비용 | 있음 (API) | 없음 |
| 속도 | 느림 | 빠름 (수 초) |
| 반복 실험 | 부적합 | 적합 |
| 편향 가능성 | LLM 편향 | 키워드 정의 편향 |

```python
# 키워드 매칭 예시
{
    "query": "오메가3의 심혈관 건강 효능은?",
    "keywords": ["오메가3", "omega-3", "omega3", "epa", "dha", "심혈관", "fish oil"]
}
# 문서 내 키워드 하나라도 포함 → Hit = 1 (대소문자 무시)
```

### Q2. "QA 평가를 어떻게 했나요?"

**답변:**

Supabase `documents_v2`에서 카테고리별로 균등 샘플링해 총 42개 청크를 뽑았습니다.

각 청크에 대해 GPT-4o-mini로 "이 청크가 직접 답할 수 있는 질문"을 생성해서 `(질문, 정답_chunk_id)` 쌍의 데이터셋을 만들었습니다(`data/generated/qa_dataset.json`).

이 데이터셋으로 RAG 검색을 실행하고, 정답 chunk_id가 Top-K 안에 포함됐는지로 Hit Rate와 MRR을 계산했습니다.

키워드 매칭 방식과 달리 정확한 정답이 있어 평가 신뢰도가 더 높지만, 18,682개의 조밀한 코퍼스에서 특정 청크 하나를 정확히 맞추는 것은 근본적으로 어렵기 때문에 Hit@5 = 0.095로 낮게 나왔습니다. 이는 시스템 문제가 아닌 평가 설계상의 한계입니다.

---

### Q3. "왜 임베딩 모델을 바꿨나요?"

**답변:**

실험 1~4에서 17개 질문 중 5개가 임베딩 모델을 바꾸지 않는 한 개선되지 않는 것을 확인했기 때문입니다.

구체적인 문제는 두 가지였습니다:

1. **한/영 혼재 학술 텍스트 한계**: 데이터셋이 한국어 질문과 영어 논문 키워드(EPA, DHA, probiotics, curcumin 등)가 섞인 구조인데, `e5-small`(118M 파라미터)은 이런 혼재 상황에서 의미 표현이 약했습니다.

2. **청크 크기 문제**: 400토큰짜리 청크는 내용이 너무 많이 담겨 임베딩이 의미를 분산해서 표현합니다. 멘토 권장 범위(150~200토큰)로 줄이면 각 벡터가 더 집중된 의미를 가집니다.

`bge-m3`는 570M 파라미터, 1024차원으로 e5-small 대비 4.8배 더 많은 파라미터를 가지며, 다국어(한/영 혼재) 학술 텍스트에 특화되어 있습니다.

결과적으로 Hit@5가 0.706 → 0.941로 +23.5%p 향상되어 임베딩 모델 교체가 옳은 판단이었음을 수치로 확인했습니다.

## 9. 다음 개선 방향

### 단기 (바로 적용 가능)

1. **QA 프롬프트 개선**: "이 텍스트", "이 연구" 같은 문맥 참조 표현 금지 명시. 사용자가 실제로 챗봇에 물어볼 법한 형태로 질문 재구성.

2. **문맥 의존 질문 자동 필터링**: 생성된 질문에서 "이", "해당", "위" 등 지시대명사 자동 탐지 및 제거.

3. **더 많은 QA 쌍 생성**: 현재 42개 → 200개 이상으로 확대 (통계적 신뢰도 향상).

### 중기

4. **카테고리별 세부 성능 분석**: 어떤 카테고리에서 실패율이 높은지 분석 후 해당 카테고리 데이터 보강.

5. **답변 생성 품질 평가**: 검색 단계(Retrieval) 외에 생성 단계(Generation) 품질 평가. RAGAS(Retrieval Augmented Generation Assessment) 프레임워크 도입 검토.
   - Faithfulness (답변이 문서 내용에 근거하는가)
   - Answer Relevancy (답변이 질문에 관련 있는가)
   - Context Precision/Recall

### 장기

6. **Parent-Child 청킹**: 작은 청크(200토큰)로 검색하되, LLM에는 더 큰 부모 청크를 전달해 컨텍스트 풍부도 향상.

7. **에이전트 구조 도입**: 단순 RAG를 넘어 멀티스텝 추론이 필요한 질문 처리 (예: "오메가3와 비타민D를 같이 먹어도 되나요?" — 두 카테고리 동시 검색 필요).
