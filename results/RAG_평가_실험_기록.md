## 1. 이 문서의 목적

이 문서는 다음 세 가지를 모두 기록합니다.

| 항목 | 내용 |
|------|------|
| 무엇을 했는가 | RAG 챗봇의 검색 품질을 정량적으로 측정하는 평가 스크립트 작성 및 4가지 실험 수행 |
| 왜 했는가 | 챗봇이 "데이터에 있는데 못 답하는" 문제의 원인을 수치로 찾기 위해 |

> **포트폴리오 활용 방법**: 이 문서 + 실험 결과 CSV를 PPT에 붙이면
> "문제 발견 → 원인 분석 → 실험 → 수치 기반 개선"의 흐름을 보여줄 수 있습니다.

## 2. 용어 정리 — 멘토 질문 먼저 이해하기

### 멘토가 물어본 것: "유사도 평가를 어떤 식으로 했냐? LLM으로 한 거냐?"

이 질문의 의도와 답변을 정확히 이해해야 합니다.

---

#### 질문의 의도

RAG 시스템에서 "검색 결과가 좋은지 나쁜지"를 판단하는 방법은 크게 두 가지입니다.

```
방법 A: LLM 기반 평가 (LLM-as-Judge)
  → LLM(GPT 등)에게 "이 문서가 질문에 답할 수 있나요? 1~5점으로 평가해" 라고 물어봄
  → 장점: 의미적으로 정교한 판단 가능
  → 단점: API 비용 발생, 평가 자체가 느리고 LLM 편향 가능성

방법 B: 규칙 기반 평가 (키워드 매칭 + 벡터 점수)
  → 사전에 정의한 "이 질문에는 이 키워드가 있어야 한다"는 기준으로 판단
  → 장점: 빠르고 무료, 반복 실험에 적합
  → 단점: 키워드 없이 의미만 맞는 문서는 놓칠 수 있음
```

멘토는 이 프로젝트가 **어떤 방법을 썼는지** 물어본 것입니다.

---

#### 이 프로젝트의 답변 (올바른 답)

> "LLM을 사용하지 않았습니다.  
> 각 질문마다 사전에 정의한 핵심 키워드 목록을 만들고,  
> 검색 결과 문서에 그 키워드가 포함되어 있으면 '관련 있음'으로 판정했습니다.  
> 이를 기반으로 Hit Rate와 MRR을 계산했습니다.  
> LLM 기반 평가는 비용과 속도 문제로 제외했고, 반복 실험이 목적이었기 때문에  
> 빠르고 재현 가능한 키워드 매칭 방식을 선택했습니다."

---

#### 두 방법의 비교

| 항목 | LLM-as-Judge | 키워드 매칭 (이 프로젝트) |
|------|-------------|--------------------------|
| 정확도 | 높음 | 보통 |
| 비용 | 있음 (API) | 없음 |
| 속도 | 느림 | 빠름 |
| 반복 실험 적합성 | 낮음 | 높음 |
| 편향 가능성 | LLM 편향 있음 | 키워드 정의에 따라 달라짐 |

> 멘토링 수준에서는 "LLM 없이 키워드 기반으로 했다, 이유는 반복 실험 목적이기 때문"  
> 이라고 답하면 충분합니다.

---

### 주요 지표 용어 설명

#### Hit Rate @K (히트율)

```
정의: 상위 K개 결과 중에 관련 문서가 1개라도 있는지 여부
계산: 관련 문서가 있으면 1, 없으면 0  →  전체 질문 평균
예시:
  질문: "오메가3 효능?" → 키워드: ["오메가3", "EPA", "DHA"]
  Top5 결과 중 2번째 문서에 "EPA, DHA" 포함  → Hit@5 = 1
  Top1 결과에는 없음                           → Hit@1 = 0
```

| Hit@1 | Hit@3 | Hit@5 |
|-------|-------|-------|
| 1등 결과가 관련 있는가 | 1~3등 중 하나라도 관련 있는가 | 1~5등 중 하나라도 관련 있는가 |
| 가장 엄격 | 중간 | 가장 느슨 |

#### MRR (Mean Reciprocal Rank, 평균 역순위)

```
정의: 첫 번째 관련 문서의 순위 역수를 평균
계산:
  1등에서 발견 → 점수 = 1/1 = 1.0
  2등에서 발견 → 점수 = 1/2 = 0.5
  3등에서 발견 → 점수 = 1/3 = 0.33
  못 찾음      → 점수 = 0

의미: 숫자가 높을수록 관련 문서가 더 앞에 나옴
```

#### Combined Score / Rerank Score

```
Combined Score: 벡터 유사도 + 키워드(BM25) 점수를 가중 합산한 값 (0~1)
  → Supabase hybrid_search RPC가 반환하는 값

Rerank Score: Combined Score + 질문 키워드 보너스(최대 +0.2)
  → 검색 후 Python에서 재정렬할 때 쓰는 점수
```

## 3. 시스템 구조 요약

### 현재 챗봇 파이프라인 (`chatbot_v2.py`)

```
사용자 질문
    ↓
[1] Query Rewrite + Multi-Query (GPT-4o-mini로 검색 쿼리 3개 생성)
    ↓
[2] Hybrid Search (벡터 검색 0.7 + BM25 키워드 검색 0.3)
    Supabase pgvector에서 TOP-10 문서 검색
    ↓
[3] Reranking (키워드 보너스 방식)
    검색 점수 + 질문 키워드 등장 횟수 × 0.05 (최대 +0.2)
    → TOP-3만 남김
    ↓
[4] LLM 답변 생성 (GPT-4o-mini)
    참고 자료로 TOP-3 문서 내용 첨부
    ↓
사용자에게 답변 반환
```

### 임베딩 모델

| 항목 | 값 |
|------|-----|
| 모델명 | `intfloat/multilingual-e5-small` |
| 파라미터 수 | 118M |
| 임베딩 차원 | 384 |
| 특징 | 한국어/영어 모두 지원, 로컬 실행 무료 |
| 한계 | 소형 모델, 한/영 혼재 학술 텍스트에서 의미 손실 가능 |

### 리랭커 현황

`chatbot_v2.py`에 `CrossEncoder("cross-encoder/ms-marco-MiniLM-L-6-v2")`가 임포트되어 있으나
실제 `rerank_documents()` 함수는 **크로스인코더를 사용하지 않고** 키워드 보너스 방식만 사용합니다.

```python
# chatbot_v2.py의 실제 rerank_documents 함수
doc["rerank_score"] = float(search_score) + keyword_bonus  # cross-encoder 미사용!
```

이 점이 **실험 4의 출발점**입니다.

## 4. 평가 설계

### 4.1 평가 방법론 선택 이유

**LLM 기반 평가를 쓰지 않은 이유:**
- 실험을 3~5회 반복해야 하는데 LLM 호출 비용이 누적됨
- 빠른 피드백 루프가 필요 (실험 하나당 수 초 완료)
- 키워드 매칭으로도 검색 실패/성공의 대부분을 잡아낼 수 있음

**Hit Rate + MRR을 선택한 이유:**
- 검색 시스템 평가의 국제 표준 지표 (Information Retrieval 분야 기준)
- Hit Rate: "찾긴 하는가?" (존재 여부)
- MRR: "얼마나 앞에서 찾는가?" (순위 품질)
- 두 지표를 같이 보면 "찾는데 순위가 낮다" vs "아예 못 찾는다"를 구분 가능

### 4.2 테스트 케이스 구성 (18개)

| 카테고리 | 질문 수 | 예시 |
|---------|---------|------|
| 건강식품 (오메가3, 프로바이오틱스, 커큐민) | 3 | "오메가3의 심혈관 건강 효능은?" |
| 다이어트 (단식, 케토, 저탄, 단백질, 섬유) | 5 | "간헐적 단식 16:8 방법과 체중 감량 효과" |
| 혈당 (스파이크, GI 지수) | 2 | "혈당 스파이크 원인과 예방법" |
| 푸드올로지 제품 (버닝올로지, 콜레올로지, 맨올로지) | 3 | "푸드올로지 버닝올로지 성분" |
| 실생활 (닭가슴살, 보조제, 보충제) | 3 | "닭가슴살 다이어트 식단 추천" |
| 비타민C (DB 미보유 가능성 테스트) | 1 | "비타민C 결핍 증상" |
| 주제 외 (폴백 확인) | 1 | "날씨가 맑은 날 운동하면 좋은가요" |

- **판정 대상 (judge=True)**: 17개
- **판정 제외**: 1개 (주제 외 질문)

### 4.3 키워드 기반 관련성 판정 원칙

각 질문에 대해 관련 문서가 포함해야 할 키워드를 사전에 정의합니다.

```python
# 예시
{"query": "오메가3의 심혈관 건강 효능은?",
 "keywords": ["오메가3", "omega-3", "omega3", "epa", "dha", "심혈관", "fish oil"]}
```

문서 내 키워드 포함 여부: **대소문자 무시 (case-insensitive)**  
판정: 키워드 중 **하나라도** 포함되면 "관련 있음"

### 실험 2 의도 (가중치 변경)

현재 벡터 검색이 70%, 키워드 검색이 30%입니다.  
한국어 질문에서 키워드 검색의 비중을 높이면 더 정확한 결과가 나올지 확인합니다.

### 실험 3 의도 (TOP_K 확대)

검색 범위를 5개→10개로 늘리면, 현재 6~10등 안에 있는 관련 문서를  
Hit@5 계산 범위 안으로 끌어올 수 있는지 확인합니다.

### 실험 4 의도 (BGE 리랭커)

현재 리랭커는 키워드 보너스 방식(영어 전용 CrossEncoder 로드 후 미사용)입니다.  
`BAAI/bge-reranker-v2-m3`는 **한국어/영어 모두 지원**하는 크로스인코더입니다.  
동일한 검색 결과를 더 정확하게 순위를 매겨 Hit@1과 MRR을 높일 수 있는지 확인합니다.

> **크로스인코더란?**  
> 일반 임베딩은 질문과 문서를 각각 벡터로 변환 후 코사인 유사도를 계산합니다.  
> 크로스인코더는 (질문, 문서) 쌍을 함께 입력받아 관련성 점수를 직접 계산합니다.  
> 더 정확하지만 느립니다 (문서 수만큼 모델을 반복 실행해야 하므로).

### 실험 5 의도 (임베딩 모델 교체 + 재청킹)

실험 1~4에서 5개 질문이 꾸준히 실패하는 근본 원인을 분석한 결과:  
- 임베딩 유사도가 낮아 관련 문서가 Top10에도 진입 못 함  
- 청크 크기(400토큰)가 커서 핵심 내용이 희석됨

**두 가지를 동시에 변경:**
1. 임베딩 모델: `e5-small(384차원)` → `bge-m3(1024차원)` (한/영 혼재 학술 텍스트 특화)
2. 청크 크기: 400토큰 → 200토큰, 오버랩 60→40 (더 조밀한 청킹)
3. Supabase에 별도 테이블 `documents_v2` 생성 (기존 `documents` 유지로 롤백 가능)
4. 청크 수: 8,760개 → 18,682개 (2.1배 증가)

> **두 변수를 동시에 바꾼 이유:**  
> 재임베딩을 하려면 재청킹도 해야 하기 때문 (이미 배포된 시스템에서 청크 크기 변경 = 전체 재업로드).  
> 재청킹 비용이 발생하는 시점에 최적 청크 크기(200토큰)도 함께 적용하는 것이 효율적.

### 실험 6, 7 의도 (BM25 가중치 탐색)

bge-m3 임베딩 전환 후 BM25 키워드 검색의 비중을 높였을 때 효과가 있는지 확인.

- 실험 6: vector 0.6 / keyword 0.4
- 실험 7: vector 0.5 / keyword 0.5

### 6.2 실험별 상세 분석

#### 실험 1: Baseline

```
Hit@1 = 0.471   → 17개 중 8개 질문이 1등 결과에서 바로 관련 문서 찾음
Hit@3 = 0.647   → 17개 중 11개 질문이 1~3등 안에서 관련 문서 찾음
Hit@5 = 0.706   → 17개 중 12개 질문이 1~5등 안에서 관련 문서 찾음
MRR   = 0.564   → 평균적으로 1.8등에서 첫 관련 문서 발견
```

**성공 질문 (12개):** 오메가3, 간헐적 단식, 케토제닉, 저탄수화물, 단백질, 혈당 스파이크, GI 지수, 푸드올로지 3종, 다이어트 보조제, 단백질 보충제

**실패 질문 (5개):**
- 프로바이오틱스 (키워드가 Top5 안에 없음)
- 커큐민 (키워드가 Top5 안에 없음)
- 식이섬유와 장내 미생물 (키워드가 Top5 안에 없음)
- 닭가슴살 다이어트 (키워드가 Top5 안에 없음)
- 비타민C 결핍 (DB에 데이터 없을 가능성)

---

#### 실험 2: 가중치 50/50

```
Hit@1 = 0.471   → Baseline과 동일 (개선 없음)
MRR   = 0.564   → Baseline과 동일
평균유사도 = 0.455  → Baseline 0.637에서 크게 하락 ❌
```

**결론: 가중치 변경은 효과 없음, 오히려 유사도 점수 하락**

이유 분석:
- `combined_score`는 `vector_weight × 벡터점수 + text_weight × BM25점수`로 계산됩니다.
- 벡터 가중치를 0.7→0.5로 낮추면 `combined_score` 자체의 절대값이 낮아집니다.
- 순위(어떤 문서가 상위에 오는지)는 변하지 않았지만 점수 숫자가 낮아졌습니다.
- **가중치 변경은 검색 결과 순서를 바꾸지 못했습니다** — 즉 이 케이스의 문제는 가중치가 아닙니다.

---

#### 실험 3: TOP_K=10

```
Hit@1 = 0.471   → Baseline과 동일
Hit@5 = 0.706   → Baseline과 동일 (Top5는 그대로)
MRR   = 0.578   → Baseline 0.564에서 소폭 향상 (+0.014)
평균유사도 = 0.653  → Baseline 0.637에서 향상 (+0.016)
```

**발견:** 프로바이오틱스는 Top10에서 **8등**, 커큐민은 **9등**에서 발견됨
→ 검색 범위를 늘리면 이 두 문서는 범위 안에 들어옴
→ 그러나 여전히 Top5 밖 → Hit@5는 개선 안 됨
→ **문제는 "검색 범위 부족"이 아니라 "임베딩 유사도가 낮아 순위 자체가 낮음"**

---

#### 실험 4: BGE 리랭커 (`BAAI/bge-reranker-v2-m3`)

```
Hit@1 = 0.588   → Baseline 0.471에서 +0.117 향상 ✅ 가장 큰 개선
Hit@3 = 0.647   → Baseline과 동일
Hit@5 = 0.706   → Baseline과 동일
MRR   = 0.629   → Baseline 0.564에서 +0.065 향상 ✅
```

**결론: BGE 리랭커가 가장 효과적**

변화가 발생한 질문:
- `단백질 섭취량과 근육 유지 관계`: 기존 Hit@1=0 → BGE Hit@1=1 (관련 문서를 1등으로 올림)
- `단백질 보충제 먹는 방법`: 기존 Hit@1=0 → BGE Hit@1=1 (동일)

이유 분석:
- 키워드 보너스 방식은 질문 단어가 문서에 그대로 들어있어야 점수를 줍니다.
- BGE는 (질문, 문서) 쌍의 **의미적 관련성**을 직접 계산하므로 한국어에서 더 정확합니다.
- 특히 "단백질"처럼 질문과 문서 용어가 일치하지 않아도 관련성을 찾아냅니다.

---

### 6.3 여전히 실패하는 5개 질문 분석

모든 실험(1~4)에서 공통으로 Hit@5 = 0인 질문:

| 질문 | 추정 원인 |
|------|----------|
| 프로바이오틱스가 장 건강에 미치는 영향 | Top10에는 있음(8등). 임베딩 유사도가 낮아 앞 순위로 못 올라옴 |
| 커큐민의 항염 효과 임상 연구 | Top10에는 있음(9등). 동일 원인 |
| 식이섬유와 장내 미생물 관계 | Top10에도 없음. 청킹 문제 가능성 (관련 내용이 두 청크에 걸쳐 있을 수 있음) |
| 닭가슴살 다이어트 식단 추천 | Top10에도 없음. 데이터에 관련 내용 부족 가능성 |
| 비타민C 결핍 증상 | DB에 비타민C 관련 데이터가 없을 가능성 높음 |

**진단 결론:**
- 검색 자체는 모두 성공 (결과 0건인 질문 없음)
- 문제 위치: `[임베딩 검색]` 단계 — 관련 문서의 유사도 점수 자체가 낮음
- 해결 방향: 임베딩 모델 교체 (`BAAI/bge-m3`) 또는 청킹 전략 개선

---

#### 실험 5: BGE-M3 임베딩 + 청킹 200토큰

```
Hit@1 = 0.765   → 실험 4(0.588) 대비 +0.177 향상 ✅ 가장 큰 도약
Hit@3 = 0.882   → 실험 4(0.647) 대비 +0.235 향상 ✅
Hit@5 = 0.941   → 실험 4(0.706) 대비 +0.235 향상 ✅
MRR   = 0.828   → 실험 4(0.629) 대비 +0.199 향상 ✅
```

**결론: 임베딩 모델 교체 + 재청킹이 가장 결정적인 개선**

- 실험 1~4에서 꾸준히 실패했던 프로바이오틱스, 커큐민이 Top5 이내로 진입
- 청크 크기를 절반(400→200토큰)으로 줄이면서 핵심 내용 밀도 향상
- bge-m3가 한/영 혼재 학술 텍스트에서 e5-small 대비 더 정확한 벡터 생성

> ※ 등급이 C로 표시되는 이유: 평균유사도가 0.420으로 임계값(0.5) 미달.  
> 이는 bge-m3의 combined_score 스케일이 e5-small과 다르기 때문 (실제 의미 없음).

---

#### 실험 6, 7: BM25 가중치 변경 (60/40, 50/50)

```
Hit@1 = 0.765   → 실험 5와 동일
Hit@3 = 0.882   → 실험 5와 동일
Hit@5 = 0.941   → 실험 5와 동일
MRR   = 0.828   → 실험 5와 동일
```

**결론: BM25 가중치 변경은 효과 없음 (bge-m3 환경에서)**

- 실험 2(e5-small 환경)와 동일한 패턴: 순위 변화 없이 점수 스케일만 바뀜
- bge-m3의 벡터 검색이 이미 충분히 정확하여 BM25 비중을 높여도 순위가 변하지 않음
- **결론: 실험 5(vector 0.7 / BM25 0.3) 설정이 최적**

## 7. 원인 진단

### 7.1 진단 프레임워크

```
질문 → [임베딩 검색] → [리랭킹] → [LLM 답변]
         ↑ 여기가 문제    ↑ 개선됨     ↑ 미평가
```

### 7.2 각 단계별 진단 결과

| 단계 | 상태 | 근거 |
|------|------|------|
| 임베딩 검색 | ⚠️ 개선 필요 | 5개 질문에서 Top10 안에 관련 문서 없음 또는 순위 낮음 |
| 리랭킹 | ✅ 개선 확인 | BGE 교체 후 Hit@1 +11.7%, MRR +6.5% |
| LLM 답변 | 미평가 | 이번 실험 범위 밖 |

### 7.3 리랭킹 단계 개선의 의미

실험 4에서 Hit@1이 크게 향상된 것은:
- 동일한 문서들이 검색되었지만
- BGE가 더 정확하게 관련 문서를 1~2등으로 올려줬기 때문

즉, 기존 키워드 보너스 방식의 문제:
- "단백질 보충제" 질문에서 "단백질"이 들어있는 문서라면 모두 보너스를 받음
- 하지만 BGE는 질문의 의도(보충제 복용 방법)와 문서 내용의 실제 관련성을 함께 판단

### 7.4 현재 `ms-marco-MiniLM-L-6-v2`의 문제

`chatbot_v2.py`에는 이 모델이 임포트되어 있지만 **실제 사용하지 않습니다**.  
또한 이 모델은 **영어 전용** 데이터로 학습된 모델입니다.  
한국어 질문에 적용하면 오히려 순위를 잘못 매길 가능성이 있습니다.  
실험 4에서 한국어/영어 모두 지원하는 BGE로 교체한 것이 효과적이었습니다.
