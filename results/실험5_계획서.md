## 2. "재임베딩"이란 무엇인가 — 개념 정리

이 질문이 핵심입니다. **재임베딩과 재청킹은 다릅니다.**

```
[원본 문서] → [청킹] → [임베딩] → [Supabase 저장]
              ↑           ↑
           재청킹       재임베딩
```

### 재임베딩만 하는 경우

```
기존 청크(텍스트) 그대로 유지
임베딩 모델만 교체: e5-small(384차원) → bge-m3(1024차원)
새 벡터를 Supabase에 업로드
```

- 무엇이 바뀌나: 각 청크의 벡터 숫자만 달라짐
- 청크 내용(텍스트)은 그대로
- 빠름 (파싱 스킵)
- 단점: 기존 청킹 전략의 문제는 해결 안 됨

### 재청킹 + 재임베딩 (이번 실험 5에서 할 것)

```
원본 문서 → 새 청킹 설정으로 다시 자르기 → 새 임베딩 모델로 벡터 생성 → Supabase 업로드
```

- 무엇이 바뀌나: 청크 텍스트 자체 + 벡터 모두 달라짐
- 느림 (전체 파싱부터 다시)
- 장점: 청킹 전략 문제까지 동시에 해결 가능

### 이번 실험에서 재청킹도 하는 이유

실험 1~4 진단에서 5개 질문이 계속 실패한 원인 중 하나가 **청킹 문제** 가능성이 있습니다.

현재 청킹 설정:
```python
CHUNK_TOKENS   = 400   # 청크 크기 (토큰 단위)
OVERLAP_TOKENS = 60    # 겹치는 부분
```

멘토 권장 범위:
```
150 ~ 200 tokens
```

현재 400 토큰짜리 청크는 내용이 너무 많이 담겨서 임베딩이 의미를 잘 못 잡을 수 있습니다.
작은 청크일수록 각 벡터가 더 집중된 의미를 가집니다.

```
[원본 데이터] (동일)
       ↓
[1단계] parse_all_data.py (청킹 파라미터 변경)

  - 청킹: CHUNK_TOKENS = 200  (400 → 200으로 축소)
           OVERLAP_TOKENS = 40  (60 → 40으로 축소)
  - 출력: data/generated/chunks/all_chunks_v2.json  (새 파일명)
       ↓
[2단계] upload_to_supabase_v2.py (임베딩 모델 변경)
  - 임베딩 모델: BAAI/bge-m3 (1024차원)
  - 프리픽스: 없음 (bge-m3는 프리픽스 불필요)
  - Supabase documents_v2 테이블에 INSERT  ← 새 테이블 필요!
       ↓
[Supabase]
  documents_v2 테이블 (새로 생성)
    - content: TEXT
    - embedding: vector(1024)   ← 차원 변경
    - category, source_file, ...
```

## 5. Supabase 테이블 구조 변경이 필요한 이유

가장 중요한 포인트입니다.

| 항목 | 현재 | 변경 후 |
|------|------|---------|
| 임베딩 차원 | 384 | 1024 |
| Supabase 컬럼 타입 | `vector(384)` | `vector(1024)` |

`vector(384)` 컬럼에 1024차원 벡터를 넣으면 **오류**가 납니다.  
따라서 두 가지 옵션이 있습니다.

### 옵션 A: 기존 테이블 삭제 후 재생성 (위험)

```sql
DROP TABLE documents;
CREATE TABLE documents (
  embedding vector(1024),
  ...
);
```

- 장점: 깔끔
- 단점: 기존 데이터 완전 삭제, 되돌리기 불가

### 옵션 B: 새 테이블 생성 (권장) ✅

```sql
CREATE TABLE documents_v2 (
  embedding vector(1024),
  ...
);
```

- 장점: 기존 `documents` 테이블 유지 → 실험 5 실패해도 롤백 가능
- 단점: 테이블이 두 개 생김
- **이 방법을 권장합니다**

## 6. 단계별 실행 계획

### Phase 0: 사전 준비 (실험 5 브랜치 시작 전)

```
① chatbot_v2.py BGE 리랭커 반영 (현재 브랜치)
② feature/rag-evaluation → main merge
③ origin/main push
④ git checkout -b feature/embedding-upgrade
```

### Phase 1: Supabase 새 테이블 생성

Supabase SQL Editor에서 직접 실행:

```sql
-- 1. 새 테이블 생성 (1024차원)
CREATE TABLE documents_v2 (
  id          BIGSERIAL PRIMARY KEY,
  source_file TEXT,
  category    TEXT,
  description TEXT,
  page_number INT,
  chunk_index INT,
  content     TEXT,
  token_count INT,
  embedding   vector(1024),
  metadata    JSONB
);

-- 2. 벡터 검색 인덱스
CREATE INDEX ON documents_v2
USING ivfflat (embedding vector_cosine_ops)
WITH (lists = 100);

-- 3. hybrid_search 함수도 documents_v2를 보도록 새로 만들어야 함
--    (기존 hybrid_search는 documents 테이블 기준)
```

### Phase 2: 청킹 파라미터 변경

`parse_all_data.py`에서 아래 두 줄만 변경합니다.

```python
# 변경 전
CHUNK_TOKENS   = 400
OVERLAP_TOKENS = 60

# 변경 후
CHUNK_TOKENS   = 200
OVERLAP_TOKENS = 40
```

그리고 출력 파일명도 변경:
```python
OUTPUT_FILE = OUTPUT_DIR / "all_chunks_v2.json"  # _v2 추가
```

실행:
```bash
python src/parse_all_data.py
```

결과: `data/generated/chunks/all_chunks_v2.json` 생성

### Phase 3: 임베딩 모델 교체 + 업로드

`upload_to_supabase.py` 수정:

```python
# 변경 전
EMBED_MODEL_NAME = "intfloat/multilingual-e5-small"
EMBED_DIMENSION  = 384
TABLE_NAME       = "documents"

# 변경 후
EMBED_MODEL_NAME = "BAAI/bge-m3"
EMBED_DIMENSION  = 1024
TABLE_NAME       = "documents_v2"
```

bge-m3는 e5와 달리 프리픽스 없이 텍스트 그대로 입력:
```python
# e5-small 방식 (변경 전)
prefixed = [f"passage: {t}" for t in texts]

# bge-m3 방식 (변경 후)
prefixed = texts  # 프리픽스 불필요
```

실행:
```bash
python src/upload_to_supabase.py
# 약 570MB 모델 다운로드 (최초 1회)
# 전체 문서 재임베딩 + 업로드
```

### Phase 4: chatbot_v2.py 수정 (테스트용)

```python
# 변경 전
embed_model = SentenceTransformer("intfloat/multilingual-e5-small")
TABLE_NAME = "documents"  # hybrid_search RPC 내부 테이블명

# 변경 후
embed_model = SentenceTransformer("BAAI/bge-m3")
TABLE_NAME = "documents_v2"
```

## 7. 실험 5에서 기대하는 결과

### 기대 효과

| 지표 | 실험 4 (현재 최고) | 실험 5 (목표) |
|------|----------------:|------------:|
| Hit@1 | 0.588 | 0.650+ |
| Hit@5 | 0.706 | 0.800+ |
| MRR | 0.629 | 0.700+ |

### 기대 근거

1. **청킹 200 토큰**: 각 청크가 더 집중된 의미 → 임베딩이 더 정확한 벡터 생성
2. **bge-m3**: 570M 파라미터, 1024차원 → e5-small(118M, 384차원) 대비 훨씬 풍부한 표현
3. **한/영 혼재 학술 텍스트 강점**: 논문 키워드(EPA, DHA, probiotics 등)와 한국어 질문 사이의 의미를 더 잘 연결

### 가장 큰 기대 질문

현재 계속 실패하는 5개 중:

| 질문 | 실패 원인 추정 | 기대 결과 |
|------|--------------|----------|
| 프로바이오틱스 장 건강 | e5-small에서 유사도 낮음 (Top10에서 8등) | bge-m3에서 Top5 진입 기대 |
| 커큐민 항염 효과 | 동일 (Top10에서 9등) | 동일 기대 |
| 식이섬유와 장내 미생물 | 청킹 경계 문제 가능성 | 200토큰 청킹으로 개선 기대 |
| 닭가슴살 다이어트 | 데이터 부족 가능성 | 개선 미지수 |
| 비타민C 결핍 | DB에 데이터 없을 가능성 | 개선 어려움 |

## 8. 위험 요소 및 대응

| 위험 | 발생 가능성 | 대응 |
|------|-----------|------|
| Supabase 기존 데이터 삭제 | 낮음 (새 테이블 사용) | documents_v2 분리로 방지 |
| bge-m3 로컬 RAM 부족 | 중간 (570MB 모델) | 배치 크기 축소로 대응 |
| 재임베딩 시간 과다 | 낮음 (로컬 처리) | 수 분~수십 분 예상 |
| 성능이 오히려 하락 | 낮음 | documents 테이블 유지로 롤백 가능 |
| Supabase hybrid_search 함수 수정 필요 | 높음 | Phase 1에서 새 RPC 함수 등록 |
