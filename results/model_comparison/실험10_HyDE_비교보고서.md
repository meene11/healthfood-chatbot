## HyDE란?

```
[기존 방식]
사용자 질문 → 임베딩 → 벡터 검색

[HyDE 방식]
사용자 질문 → LLM → 가상 답변(논문 스타일 생성) → 임베딩 → 벡터 검색
                                                    ↑ 이 임베딩으로 검색
BM25 키워드 검색은 원본 질문 그대로 사용
```

**핵심 아이디어**: 한국어 질문을 영어 논문 청크와 직접 매칭하는 대신,  
LLM이 먼저 "논문에 이런 내용이 있을 것"이라는 가상 문서를 만들어 그 임베딩으로 검색 → 표현 갭 해소

## 2. 핵심 분석

### 2-1. GPT-4o-mini: 답변 품질 대폭 향상 ★

- 품질: 1.048 → **1.262** (↑ +0.214, 가장 큰 향상)
- 완전정답(3점): 19.0% → **26.2%** (+7.2%)
- Hit@5는 소폭 감소(0.095 → 0.071)했으나 답변 품질은 크게 올라감
  → 검색된 청크가 달라졌지만 더 관련성 높은 내용을 찾은 것으로 해석

### 2-2. Llama 3.3 70B: 검색 품질 획기적 개선 ★

- Hit@5: 0.095 → **0.190** (↑ +0.095, 2배 향상)
- MRR: 0.048 → **0.151** (↑ +0.103, 3배 향상)
- 답변 품질은 소폭 향상(0.595 → 0.619)에 그침
  → 검색은 잘 되지만 Llama 자체의 답변 생성 능력 한계가 병목

### 2-3. Gemini Flash Lite: 전반적 향상이나 속도 문제 ★

- 품질: 1.262 → **1.381** (↑ +0.119)
- Hit@5: 0.095 → **0.167** (↑ +0.072)
- **응답 시간: 7.8초 → 19.8초** (2.5배 증가, 사용자 경험 저하 우려)

### 2-4. HyDE 오버헤드 분석

HyDE는 매 질문마다 GPT-4o-mini로 가상 문서를 생성하는 추가 API 호출이 발생:
- GPT, Llama: +2.5~3.4초 — 허용 가능한 수준
- Gemini: +12.0초 — Gemini 자체 응답 지연과 합산되어 실사용 어려운 수준
