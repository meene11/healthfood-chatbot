```

  ├─ 검색 품질: Hit@K, MRR (정답 청크를 Top-K 안에서 찾는 비율)
  └─ 답변 품질: LLM-as-Judge (GPT-4o-mini가 0~3점 채점)
```

```

       ↓                               ↓                          ↓
```

## PART 1 — 검색 파이프라인 최적화 (실험 1~7)

### 개념: 검색 품질 지표란?

| 지표 | 정의 | 예시 |
|------|------|------|
| **Hit@K** | 상위 K개 결과 안에 정답이 있는 비율 | Hit@5=0.8 → 10문항 중 8개 Top5 안에 정답 |
| **MRR** | 정답이 몇 번째에 나오는지의 역수 평균 | MRR=0.5 → 정답이 평균 2위 |
| **평균 유사도** | 검색된 문서와 질문의 cosine 유사도 | 높을수록 관련성 높음 |

> **왜 중요한가?** — 검색이 잘 되어야 LLM도 좋은 답변을 할 수 있음. 검색이 실패하면 LLM도 틀린 답변을 생성함.

---

### 실험 1 — 기준선 측정 (Baseline)

**목적:** 현재 파이프라인의 성능 기준점 확인  
**설정:** `intfloat/multilingual-e5-small` (384-dim) + 벡터:키워드 = 70:30  
**평가:** 검색 품질 18개 테스트 케이스

| 지표 | 값 |
|------|-----|
| Hit@5 | 0.548 |
| MRR | 0.412 |
| 평균 유사도 | 0.52 |
| 등급 | **C (보통)** |

---

### 실험 2~3 — 하이브리드 검색 가중치 / Top-K 튜닝

**목적:** 벡터:키워드 비율과 검색 개수 최적화  
**실험한 조합:**

| 실험 | 벡터 | 키워드 | Top-K | Hit@5 |
|------|:---:|:---:|:---:|:---:|
| Exp 1 (기준) | 70% | 30% | 5 | 0.548 |
| Exp 2 | 50% | 50% | 5 | 0.561 |
| Exp 3 | 70% | 30% | 10 | 0.572 |

**결론:** 가중치 변경보다 임베딩 모델 교체가 더 효과적임을 인식 → 실험 5로

---

### 실험 4 — BGE Reranker 도입

**목적:** 검색 결과 재정렬로 정밀도 향상  
**변경:** `cross-encoder/ms-marco-MiniLM-L-6-v2` 도입

```
[이전] 벡터 점수만으로 Top-3 선택
[이후] 벡터 검색 Top-10 → Reranker로 질문-문서 관련도 재평가 → Top-3 선택
```

| 지표 | 도입 전 | 도입 후 |
|------|:---:|:---:|
| Hit@3 | 0.489 | **0.611** |
| 평균 유사도 | 0.52 | **0.61** |

**핵심:** Cross-Encoder는 질문과 문서를 **함께 보고** 관련도를 판단 → Bi-Encoder(임베딩)보다 정확

---

### 실험 5~7 — BGE-M3 임베딩 모델 교체 ⭐ 핵심 개선

**목적:** 영어 논문과 한국어 질문 간 매칭 성능 대폭 향상  
**변경:** `e5-small` (384-dim) → `BAAI/bge-m3` (1024-dim)

```
왜 BGE-M3인가?
├─ 벡터 차원 384 → 1024 (2.7배 더 풍부한 표현)
├─ MTEB 벤치마크 최상위권 (검색 특화)
├─ 영어·한국어 혼용 환경에서 우수한 성능
└─ 무료 오픈소스 (비용 증가 없음)
```

**결과:**

| 지표 | e5-small | BGE-M3 | 향상 |
|------|:---:|:---:|:---:|
| Hit@5 | 0.548 | **0.781** | **+42%** |
| MRR | 0.412 | **0.652** | **+58%** |
| 평균 유사도 | 0.52 | **0.71** | **+37%** |
| 등급 | C (보통) | **A (우수)** | |

**가중치 최적 조합 (실험 6~7):**

| 실험 | 벡터:키워드 | Hit@5 | MRR |
|------|:---:|:---:|:---:|
| Exp 5 | 70:30 | 0.781 | 0.652 |
| Exp 6 | 60:40 | 0.774 | 0.641 |
| Exp 7 | 50:50 | 0.761 | 0.629 |

→ **70:30 최종 채택** (원래 설정이 최적)

## PART 2 — LLM 모델 비교 평가 (실험 8~9)

### 개념: LLM-as-Judge란?

```
생성된 답변을 평가하는 방법:
  → GPT-4o-mini가 채점관 역할
  → 정답 기준(reference)과 비교해 0~3점 채점

0점: 완전히 틀리거나 "모릅니다"
1점: 핵심 내용 일부만 포함
2점: 대부분 정확하나 약간 불완전
3점: 정확하고 완전한 답변
```

**왜 사용하나?** — 42개 문항을 사람이 수동 채점하기 어렵기 때문. GPT 채점이 인간 채점과 높은 일치도를 보임.

### 실험 8 — 조원 QA 62문항으로 3개 모델 비교

**목적:** 동일 검색 파이프라인에서 LLM만 교체했을 때 품질 차이 확인  
**비교 모델:** GPT-4o-mini vs Llama 3.3 70B (Groq 무료) vs Gemini Flash Lite (무료)  
**평가 데이터:** 조원이 만든 QA 100개 → 내 챗봇 적합 62개 필터링

| 지표 | GPT-4o-mini | Llama 3.3 70B | Gemini Flash Lite |
|------|:-----------:|:-------------:|:-----------------:|
| **평균 품질 (0~3)** | 1.581 | **★1.629** | 1.468 |
| **완전정답 (3점)** | **★35.5%** | 27.4% | 33.9% |
| **오답 (0점)** | 33.9% | **★25.8%** | 35.5% |
| **응답 속도** | 8.14초 | **★5.56초** | 14.0초 |
| **API 비용** | $0.15/1M | **★무료** | **★무료** |

```
답변 패턴 분석:
GPT-4o-mini:  ████████████ 3점 22개  █████ 0점 21개  ← 양극화 (확실히 알면 3점)
Llama 3.3:    ████████ 3점 17개  ████████████ 2점 21개  ← 안정적 (부분 정보라도 제공)
Gemini Flash: ████████████ 3점 21개  ████ 0점 22개  ← GPT와 유사 패턴
```

**결론:** Llama가 평균 품질 1위, 무료, 빠름 → 유력 대안이나 내 DB 기반 검증 필요

### 실험 9 — 내부 QA 42문항 + 검색 품질 동시 측정

**목적:** 내 documents_v2 DB 기반 질문으로 더 공정한 평가  
**추가 지표:** Hit@K, MRR (검색 품질도 동시 측정)

| 지표 | GPT-4o-mini | Llama 3.3 70B | Gemini Flash Lite |
|------|:-----------:|:-------------:|:-----------------:|
| **평균 품질 (0~3)** | 1.048 | 0.595 | **★1.262** |
| **오답률 (0점)** | 42.9% | **71.4%** | **★26.2%** |
| **Hit@5 (검색)** | 0.095 | 0.095 | 0.095 |
| **MRR (검색)** | 0.048 | 0.048 | 0.048 |

**핵심 발견:**
- 검색 지표(Hit@K, MRR)는 3개 모델 모두 동일 → 검색은 공통 파이프라인이라 LLM과 무관
- Llama는 실험 9에서 오답률 **71.4%** 로 급격히 나빠짐 → 내 DB 특화 질문에 취약
- Hit@5 = 9.5% 로 매우 낮음 → 검색 자체 개선 필요성 확인

**실험 8 vs 9 비교 (품질 변화):**

| 모델 | 실험8 | 실험9 | 변화 |
|------|:---:|:---:|:---:|
| GPT-4o-mini | 1.581 | 1.048 | ↓ -0.533 |
| Llama 3.3 70B | 1.629 | 0.595 | **↓ -1.034** |
| Gemini Flash Lite | 1.468 | 1.262 | ↓ -0.206 (가장 안정적) |

## PART 3 — 검색 알고리즘 개선 (실험 10~11)

### 개념: HyDE (Hypothetical Document Embeddings)

```
문제: 한국어 질문 → 영어 논문 청크 직접 매칭 시 표현 갭 발생

해결책:
  [기존] 질문 → 임베딩 → 검색
  [HyDE] 질문 → LLM → "논문 스타일 가상 답변" → 임베딩 → 검색
                          ↑ 이걸로 검색하면 논문 표현과 더 잘 매칭됨

BM25(키워드) 검색은 원본 질문 그대로 사용
```

### 개념: 쿼리 확장 (Query Expansion)

```
문제: 사용자가 쓰는 표현 vs 논문에서 쓰는 표현이 다름
  예) "살 빠지는 건강식품" vs "체중 감량 임상 연구 논문"

해결책: 질문 1개 → GPT가 3가지 다른 표현으로 확장
  → 각각 검색 → 결과 합산(중복 제거) → Reranker로 최선 선별
```

### 실험 11 — 쿼리 확장 효과

**결과:** 수집 문서 5개 → 평균 14.2개, GPT +0.357 / Gemini +0.190 품질 향상

| 모델 | 품질 실험9 | 품질 실험11 | 변화 | Hit@5 변화 | 응답시간 |
|------|:---:|:---:|:---:|:---:|:---:|

### 검색 품질 개선 누적

| 단계 | 변경 내용 | Hit@5 | 등급 |
|------|---------|:---:|:---:|
| 시작 | e5-small + 기본 설정 | 0.548 | C |
| Exp 4 | BGE Reranker 도입 | 0.631 | B |
| **Exp 5** | **BGE-M3 임베딩 교체** | **0.781** | **A** |
| Exp 10 | HyDE 적용 (GPT 기준) | 검색 개선 확인 | A+ |

### LLM 모델 최종 선택 가이드

| 상황 | 추천 | 근거 |
|------|------|------|
| **현재 운영 (안정성 우선)** | GPT-4o-mini | 두 실험 모두 안정적 2위, 이미 연동됨 |
| **품질 최우선** | Gemini Flash Lite | 3회 실험 평균 1.370으로 최고, 무료 |
| **속도+무료** | Llama 3.3 70B | 2.6초로 가장 빠름, 단 내부 DB에 취약 |

### 품질 개선 결론

1. **가장 큰 개선: 임베딩 모델 교체 (Exp 5)**  
   e5-small → BGE-M3로 교체만으로 Hit@5 +42%, MRR +58% 향상  
   → *"LLM을 바꾸는 것보다 임베딩 모델이 검색 품질에 더 큰 영향"*

2. **LLM 선택: 조원 QA vs 내부 QA 결과 달라**  
   - 조원 QA: Llama 1위 / 내부 QA: Gemini 1위  
   → *"평가 데이터가 실제 DB와 얼마나 유사한지가 중요"*

3. **HyDE: 코드만 바꿔 품질 향상 가능**  
   DB 변경 없이 검색 방식만 바꿔 GPT 품질 +0.214 달성  
   → *"알고리즘 개선이 인프라 비용 없이 품질 향상 가능"*
