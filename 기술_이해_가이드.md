## 2. RAG란? 이 프로젝트에서 어떻게 구현됐나?

RAG = **Retrieval-Augmented Generation** (검색 + 생성)

LLM(GPT 등)은 학습된 지식만으로 답하면 **할루시네이션(거짓말)** 이 생김.  
RAG는 **외부 문서를 실시간으로 검색해서 그걸 근거로 답변**을 만드는 방식.

```
질문 → 관련 문서 검색 → 검색 결과를 컨텍스트로 주입 → LLM 답변 생성
```

### 현재 파이프라인 (최종 버전)

| 단계 | 구현 방법 |
|------|-----------|
| 문서 저장 | Supabase pgvector (`documents_v2`, 18,682청크) |
| 임베딩 | `BAAI/bge-m3` 1024-dim (로컬, 무료) |
| 검색 | Hybrid Search (벡터 70% + BM25 30%) + HyDE 옵션 |
| 쿼리 확장 | GPT-4o-mini로 3가지 변형 쿼리 자동 생성 |
| 리랭킹 | `BAAI/bge-reranker-v2-m3` Cross-Encoder (로컬, 무료) |
| 답변 생성 | GPT-4o-mini (스트리밍) |
| 웹 UI | Streamlit (HuggingFace Spaces 배포) |

## 3. 기술 스택 (현재 버전)

| 역할 | 기술 | 비용 |
|------|------|------|
| LLM | GPT-4o-mini (OpenAI) | 질문 1회 ~0.1원 |
| 임베딩 | `BAAI/bge-m3` (1024-dim, 다국어) | 무료 (로컬) |
| 리랭킹 | `BAAI/bge-reranker-v2-m3` (Cross-Encoder) | 무료 (로컬) |
| 벡터 DB | Supabase pgvector (HNSW) | 무료 Tier |
| 검색 방식 | Hybrid (벡터 + PostgreSQL FTS) | - |
| 웹 UI | Streamlit | 무료 |
| 배포 | HuggingFace Spaces | 무료 |

### 왜 BGE-M3를 선택했나?

| 항목 | 이전 (e5-small) | 현재 (BGE-M3) |
|------|:---:|:---:|
| 벡터 차원 | 384 | **1024** |
| 다국어 지원 | 100개 언어 | **100개 언어 + 강화** |
| 영어 논문 매칭 | 보통 | **우수** |
| MTEB 벤치마크 | 중위권 | **최상위권** |
| 비용 | 무료 | **무료** |

> 실험 5: BGE-M3 전환 후 Hit@5 **0.548 → 0.781** 향상 (+42%)

### 왜 BGE Reranker v2-m3를 선택했나?

- Cross-Encoder 방식: 질문-문서 쌍을 함께 보고 관련도 판단 → 더 정확
- 이전(ms-marco MiniLM)보다 다국어 성능 우수
- 실험 4에서 BGE Reranker 도입 후 검색 정확도 유의미하게 향상

## 5. DB 스키마 (현재 — documents_v2)

이전에는 `parent_chunks` + `child_chunks` 2테이블이었으나,  
실험 결과 **단일 테이블 200토큰 청킹**이 더 효율적으로 확인되어 통합.

```sql
-- 메인 청크 테이블
CREATE TABLE documents_v2 (
    id          BIGSERIAL PRIMARY KEY,
    source_file TEXT,
    category    TEXT,         -- 건강식품_논문, 다이어트_논문, 푸드올로지 등
    content     TEXT,         -- 200토큰 분량
    embedding   VECTOR(1024), -- BGE-M3 1024-dim
    token_count INTEGER,
    metadata    JSONB
);

-- HNSW 인덱스 (빠른 벡터 검색)
CREATE INDEX ON documents_v2 USING hnsw (embedding vector_cosine_ops);

-- BM25 전체 텍스트 검색
CREATE INDEX ON documents_v2 USING gin (to_tsvector('simple', content));
```

**검색 함수:**
```sql
hybrid_search_v2(
    query_embedding  VECTOR,   -- BGE-M3 임베딩
    query_text       TEXT,     -- BM25 키워드
    match_count      INT,      -- Top-K (보통 5)
    vector_weight    FLOAT,    -- 0.7
    text_weight      FLOAT,    -- 0.3
    filter_category  TEXT      -- NULL=전체
)
```

## 6. 검색 파이프라인 상세

### 전체 흐름

```
사용자 입력: "오메가3의 심혈관 효능은?"
     ↓
[Step 1] 쿼리 확장 (Query Expansion)
         GPT-4o-mini → 3가지 변형 쿼리 생성
         → ["오메가3 심혈관 효과", "EPA DHA heart health", "오메가-3 임상 연구"]
     ↓
[Step 2] Hybrid Search (병렬 실행)
         4개 쿼리 × ThreadPoolExecutor
         BGE-M3 벡터(70%) + BM25 키워드(30%)
         카테고리 필터링 (푸드올로지 감지 시)
     ↓
[Step 3] 결과 합산 + 중복 제거
     ↓
[Step 4] BGE Reranker
         Cross-Encoder로 질문-문서 쌍 관련도 재평가
         → Top-3 선별
     ↓
[Step 5] Context 조립
         [자료 1] ~ [자료 3] + 카테고리/출처 표시
     ↓
[Step 6] 메모리뱅크 로드
         이전 대화에서 파악한 사용자 건강 정보 주입
     ↓
[Step 7] GPT-4o-mini 답변 생성 (스트리밍)
         [자료 N] 출처 표시 의무화
     ↓
답변 출력 + 메모리 업데이트
```

### HyDE (Hypothetical Document Embeddings) — 실험적 기능

```
기존: 질문 → 임베딩 → 검색
HyDE: 질문 → LLM 가상 답변 생성 → 임베딩 → 검색

효과: 한국어 질문 ↔ 영어 논문 청크 매칭률 향상
실험 10 결과: GPT-4o-mini 품질 1.048 → 1.262 (+0.214)
```

앱에서 사이드바 토글로 선택 가능.

## 7. 메모리뱅크 시스템

세션 간에 사용자 건강 정보를 기억하는 개인화 기능.

```json
{
  "facts": [
    "목표: 체중 감량 10kg",
    "알레르기: 견과류",
    "복용 중: 오메가3, 비타민D",
    "식단 방식: 간헐적 단식 16:8"
  ],
  "updated_at": "2026-04-07 14:30:00"
}
```

동작 방식:
- 대화 종료 후 GPT가 건강 관련 정보 자동 추출 → `user_memory.json` 저장
- 다음 세션 시작 시 시스템 프롬프트 앞에 자동 주입

## 8. 실험 이력 요약 (RAG 품질 개선 과정)

| 실험 | 주제 | 핵심 변화 | 주요 결과 |
|------|------|---------|---------|
| Exp 1 | 기준선 | e5-small + 기본 검색 | Hit@5=0.548 |
| Exp 2 | 가중치 조정 | 벡터:키워드 = 50:50 | Hit@5 소폭 향상 |
| Exp 3 | Top-K 증가 | Top-5 → Top-10 | Recall 향상 |
| Exp 4 | BGE Reranker | ms-marco → bge-reranker | 정밀도 향상 |
| Exp 5 | BGE-M3 임베딩 | e5-small → bge-m3 | **Hit@5 0.548→0.781 (+42%)** |
| Exp 6 | 가중치 60:40 | 벡터 60% + 키워드 40% | - |
| Exp 7 | 가중치 50:50 | 벡터 50% + 키워드 50% | 최종 70:30 채택 |
| Exp 8 | LLM 비교 (조원 QA) | GPT vs Llama vs Gemini | Llama 1.629 1위 |
| Exp 9 | LLM 비교 (내부 QA) | Hit@K + 품질 동시 측정 | Gemini 1.262 1위 |
| Exp 10 | HyDE | 가상 문서 임베딩 | GPT 품질 +0.214 향상 |
| Exp 11 | 쿼리 확장 | 원본+변형 최대 4개 검색 | 검색 문서 수 증가 |

## 9. 할루시네이션 방지 설계

### 1. 시스템 프롬프트 제약
```
"[참고 자료]에 있는 내용만으로 답변"
"관련 내용이 없으면 솔직하게 '보유한 자료에서 찾을 수 없습니다'"
"의학적 진단/처방 금지 → '전문 의사 상담을 권장'"
```

### 2. 주제 이탈 필터링
```python
OFF_TOPIC_KEYWORDS = ["날씨", "주식", "코딩", "게임", "영화", "정치", ...]
if any(kw in user_input for kw in OFF_TOPIC_KEYWORDS):
    return "해당 분야는 제 전문 영역이 아닙니다."
```

### 3. 리랭킹 점수 낮을 때 키워드 폴백
```python
LOW_RERANK_THRESHOLD = -3.0
if max_rerank_score < LOW_RERANK_THRESHOLD:
    # 키워드 직접 검색으로 보완
    fallback_docs = keyword_fallback_search(query, already_found_ids)
```
