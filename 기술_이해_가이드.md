## 2. RAG란 뭔가? 이 프로젝트에서 어떻게 구현됐나?

RAG = **Retrieval-Augmented Generation** (검색 + 생성)

LLM(GPT 등)은 학습된 지식으로만 답하면 할루시네이션(거짓말)이 생김.  
RAG는 **외부 문서를 실시간으로 검색해서 그걸 근거로 답변을 만드는 방식**이다.

```
질문 → 관련 문서 검색 → 검색 결과를 컨텍스트로 주입 → LLM 답변 생성
```

이 프로젝트는 **풀스택 RAG 파이프라인**이 모두 구현돼 있다:

| 단계 | 구현 방법 |
|------|-----------|
| 문서 저장 | Supabase (PostgreSQL + pgvector) |
| 임베딩 | intfloat/multilingual-e5-small (로컬, 무료) |
| 검색 | Hybrid Search (벡터 70% + 키워드 30%) |
| 재정렬 | 한국어 키워드 보너스 기반 리랭킹 |
| 답변 생성 | GPT-4o-mini (스트리밍) |

---파이프라인: 임베딩,llm

문서 → 임베딩 모델(e5-small)로 벡터화 → 저장
질문 → 임베딩 모델로 벡터화 → 유사 문서 검색
→ LLM(GPT-4o-mini)이 검색 결과 보고 답변 생성
----
왜 이 모델을 사용했는가?

-- 임베딩: 왜 intfloat/multilingual-e5-small을 선택했나?
① 무료 & 비용 절감
OpenAI 임베딩 모델(text-embedding-ada-002)은 API 호출할 때마다 돈이 나가. 근데 HuggingFace 오픈소스 모델은 로컬에서 무료로 돌릴 수 있어. 개인 포트폴리오 프로젝트에서 비용 부담 없이 쓸 수 있다는 게 큰 장점이야.
② 한국어 지원
multilingual이라는 이름처럼 100개 이상 언어를 지원해. 특히 한국어 임베딩 성능이 좋아서 한국어 데이터 다루는 프로젝트에 적합해. OpenAI 임베딩도 한국어 되긴 하는데, 오픈소스 중에선 e5 계열이 한국어 성능이 검증된 편이야.
③ small = 가볍고 빠름
e5 계열에는 small / base / large 버전이 있는데, small은 모델 크기가 작아서 로컬 환경에서도 빠르게 돌아가. 개인 프로젝트나 데모용으로 딱 맞아.
④ RAG에 최적화된 모델
e5 모델은 특히 검색(retrieval) 태스크에 특화되어 있어. RAG 파이프라인에서 유사 문서 찾는 용도로 설계된 모델이라 궁합이 좋아.

한 줄 요약 (발표용):

"비용 효율성과 한국어 지원, RAG 최적화를 고려해 HuggingFace의 오픈소스 다국어 임베딩 모델을 채택했습니다."

-- llm: 왜 GPT-4o-mini를 선택했나?
① 비용 대비 성능이 가장 좋음
GPT-4o (풀버전)은 성능은 좋지만 비용이 비싸. GPT-4o-mini는 풀버전 대비 훨씬 저렴하면서도 일반적인 텍스트 생성 태스크에서는 성능 차이가 크지 않아. 개인/팀 프로젝트에서 API 비용 부담을 줄이면서도 GPT-4급 품질을 쓸 수 있는 선택이야.
② 한국어 능력 우수
Claude, Gemini 등 다른 LLM도 있지만 GPT-4o-mini는 한국어 이해와 생성 품질이 안정적으로 검증되어 있어. 한국 SME 대상 마케팅 문구를 생성하는 프로젝트 특성상 한국어 품질이 중요한 기준이었어.
③ API 안정성과 생태계
OpenAI API는 문서화가 잘 되어 있고, FastAPI + LangChain 등과의 연동 레퍼런스가 많아. 개발 속도 면에서 유리하고 트러블슈팅도 쉬워.
④ 속도
mini 모델답게 응답 속도가 빨라서 실시간 스트리밍 응답 구현에 적합해.

## 5. 데이터 처리 파이프라인 (parse_and_upload_v2.py)

이게 "RAG를 위한 준비 단계"다. 챗봇을 실행하기 전에 한 번만 돌리면 됨.

### 5.1 처리 흐름

```
원본 파일 (PDF, TXT, JSON)
         ↓
[Phase 1] 파일 수집 + 텍스트 추출
         ↓
[Phase 2] Parent-Child 청킹
         ↓
[Phase 3] 임베딩 생성 (로컬, 무료)
         ↓
[Phase 4] Supabase 업로드
```

### 5.2 Parent-Child 청킹이란?

가장 핵심 기술 중 하나. 청킹 = 긴 문서를 작은 조각으로 자르는 것.

일반 청킹과 다른 점:

```
일반 청킹: 하나의 크기로만 자름 → 검색 정확도 vs 컨텍스트 풍부함 tradeoff 발생

Parent-Child 청킹:
 - Child 청크 (200토큰): 검색용 → 작아서 정밀하게 찾아냄
 - Parent 청크 (1600토큰): 답변 생성용 → 크고 풍부한 맥락 제공
```

예시로 보면:

```
원본 논문 (5,000토큰)
├─ Parent 1 (0~1600토큰)
│   ├─ Child 1 (0~200토큰)      ← 임베딩 있음, 검색 대상
│   ├─ Child 2 (160~360토큰)    ← 임베딩 있음, 검색 대상
│   ├─ Child 3 (320~520토큰)    ← 임베딩 있음, 검색 대상
│   └─ ...
├─ Parent 2 (1400~3000토큰)     ← 오버랩 200토큰으로 맥락 연결
│   └─ ...
└─ ...

→ Child가 히트되면, 해당 Child를 포함한 Parent(1600토큰)를 LLM에 전달
```

**핵심 파라미터:**
```python
PARENT_TOKENS  = 1600  # 부모 청크 크기
PARENT_OVERLAP = 200   # 부모 오버랩 (문맥 연결)
CHILD_TOKENS   = 200   # 자식 청크 크기
CHILD_OVERLAP  = 40    # 자식 오버랩
```

### 테이블 구조

```sql
CREATE TABLE parent_chunks (
    id         BIGSERIAL PRIMARY KEY,
    source_file TEXT,           -- "data/papers/diet/intermittent_fasting.pdf"

```

#### child_chunks (자식 테이블 — 검색 대상)
```sql
CREATE TABLE child_chunks (
    id          BIGSERIAL PRIMARY KEY,
    parent_id   BIGINT REFERENCES parent_chunks(id),  -- 부모 연결
    source_file TEXT,
    category    TEXT,
    page_number INTEGER,
    chunk_index INTEGER,
    content     TEXT,            -- 200토큰 분량
    token_count INTEGER,
    embedding   VECTOR(384),     -- ← 384차원 벡터, HNSW 인덱스
    metadata    JSONB,
    created_at  TIMESTAMPTZ DEFAULT NOW()
);
```

#### 인덱스
```sql
-- 벡터 유사도 검색 (HNSW 알고리즘)
CREATE INDEX ON child_chunks USING hnsw (embedding vector_cosine_ops)
WITH (m = 16, ef_construction = 64);

-- 카테고리 필터
CREATE INDEX ON child_chunks (category);

-- 전체 텍스트 검색 (BM25 대용)
CREATE INDEX ON child_chunks USING gin (to_tsvector('simple', content));
```

### Step 2: Hybrid Search

벡터 검색 + 키워드 검색을 동시에 실행하고 점수를 합산.

**왜 Hybrid?**
- 벡터 검색만: "혈당 관리" → "glucose management" 찾을 수 있지만 "오메가3" 같은 정확한 용어는 놓칠 수 있음
- 키워드만: 정확한 용어는 찾지만 "심혈관" → "heart health" 연결 못 함
- Hybrid: 둘의 장점을 결합

```sql
-- 벡터 검색 (cosine 유사도)
SELECT child_chunks
WHERE embedding IS NOT NULL
ORDER BY embedding <=> query_embedding  -- cosine distance
LIMIT 30

-- 키워드 검색 (FTS)
SELECT parent_chunks
WHERE to_tsvector('simple', content) @@ plainto_tsquery('simple', query_text)
LIMIT 30

-- 결합 점수 (Supabase RPC 함수로 구현)
combined_score = (vector_score × 0.7) + (text_score × 0.3)
```

**카테고리 자동 필터링:**
```python
def detect_query_category(query: str) -> str | None:
    if "푸드올로지" in query:
        return "푸드올로지"  # 해당 카테고리만 검색
    return None
# 결과 없으면 자동 폴백 → 전체 카테고리 재검색
```

**병렬 실행:**
```python
# 4개 워커 스레드로 멀티쿼리 병렬 처리
with ThreadPoolExecutor(max_workers=4):
    for query in search_queries:
        hybrid_search(query)
```

### Step 3: Reranking (재정렬)

Supabase가 준 점수에 한국어 키워드 보너스를 더해서 재정렬.

```python
def rerank_documents(query: str, docs: list[dict]) -> list[dict]:
    for doc in docs:
        search_score = doc.get('combined_score', 0)

        # 질문에서 2글자 이상 키워드 추출
        query_keywords = [w for w in query.split() if len(w) >= 2]

        # 문서에서 키워드 매칭 개수
        keyword_hits = sum(1 for kw in query_keywords if kw in doc['content'])
        keyword_bonus = min(keyword_hits * 0.05, 0.2)  # 최대 +0.2점

        doc['rerank_score'] = search_score + keyword_bonus

    # TOP 3만 선택
    return sorted(docs, key=lambda x: x['rerank_score'], reverse=True)[:3]
```

```json
    "식단 방식: 간헐적 단식 16:8"
  ],
  "updated_at": "2026-04-02 14:30:00"
}
```

### 동작 방식

```
[대화 종료 후]
사용자 질문 + 챗봇 답변 → GPT에게 "여기서 개인정보 추출해줘"
                        ↓
                  추출된 정보 → user_memory.json에 저장 (중복 제거)

[다음 대화 시작 시]
user_memory.json 로드 → 시스템 프롬프트 맨 앞에 주입
                      → 챗봇이 이전 맥락을 기억한 것처럼 답변
```

### 명령어
```
"메모리보기"    → 현재 저장된 정보 출력
"메모리초기화"  → 전체 삭제
```

## 11. 비용 분석

### LLM (GPT-4o-mini)
```
입력 비용:  $0.15 / 1M tokens
출력 비용:  $0.60 / 1M tokens

질문 1회당:
- 입력: 쿼리 생성(~1,000토큰) + 컨텍스트(~4,000토큰) ≈ 5,000토큰
- 출력: 답변 ~500토큰
- 비용: 약 $0.001 → 0.1원

월 10,000회: $10 (약 13,000원)
```

### 임베딩 + 리랭킹 모델
```
intfloat/multilingual-e5-small → 로컬 실행, 무료, ~130MB
cross-encoder/ms-marco         → 로컬 실행, 무료, ~80MB
```

### Supabase
```
Free tier: 500MB 저장소, 무제한 쿼리 → 무료
```

## 12. 할루시네이션 방지 설계

### 1. 시스템 프롬프트 제약
```
"반드시 [참고 자료]에 있는 내용만으로 답변"
"[참고 자료]에 없는 내용은 절대 지어내지 말 것"
"모르거나 자료 부족 → '자료에서 해당 정보를 찾을 수 없습니다'"
"추측, 일반 상식, 외부 지식 보충 금지"
"의학적 진단/처방 금지 → '전문 의사 상담을 권장'"
```

### 2. 주제 이탈 필터링
```python
OFF_TOPIC_KEYWORDS = ["날씨", "주식", "코딩", "게임", "음악", "정치", ...]

if any(kw in user_input for kw in OFF_TOPIC_KEYWORDS):
    return "해당 분야는 제 전문 영역이 아닙니다."
```

### 3. 신뢰도 평가
```python
def assess_confidence(docs: list[dict]) -> str:
    max_sim = max(doc.get("similarity", 0) for doc in docs)
    if max_sim >= 0.5:   return "high"    # 신뢰도 높음
    elif max_sim >= 0.35: return "low"    # 신뢰도 낮음
    else:                return "no_data" # 데이터 없음
```

## 14. 전체 아키텍처 다이어그램

```
┌───────────────────────────────────────────────┐
│             사용자 입력                        │
│        "오메가3의 심혈관 효능은?"             │
└──────────────────┬────────────────────────────┘
                   ↓
       ┌─────────────────────────┐
       │  [1] Query Rewriting    │ ← GPT-4o-mini (API 1회)
       │  → 검색 쿼리 3개 생성   │
       └───────────┬─────────────┘
                   ↓
   ┌───────────────┴────────────────┐
   ↓               ↓                ↓
[쿼리1]        [쿼리2]          [쿼리3]
   │               │                │
   └───────────────┼────────────────┘
       병렬 실행 (ThreadPoolExecutor)
                   ↓
       ┌─────────────────────────┐
       │  [2] Hybrid Search      │
       │  (Supabase pgvector)    │
       │                         │
       │  벡터: cosine 유사도    │
       │  키워드: FTS 검색       │
       │  결합: 70% + 30%        │
       └───────────┬─────────────┘
                   ↓
       결과 병합 + 중복 제거
                   ↓
       ┌─────────────────────────┐
       │  [3] Reranking          │
       │  검색점수 + 키워드보너스│
       │  → TOP 3 선별           │
       └───────────┬─────────────┘
                   ↓
       ┌─────────────────────────┐
       │  [4] Context Assembly   │
       │  [자료1], [자료2], [자료3]│
       └───────────┬─────────────┘
                   ↓
     메모리뱅크 로드 (이전 대화 정보 주입)
                   ↓
       ┌─────────────────────────┐
       │  [5] LLM 답변 생성      │ ← GPT-4o-mini (스트리밍)
       │  할루시네이션 방지 설계 │
       └───────────┬─────────────┘
                   ↓
     답변 출력 (실시간) + 출처 표시 + 메모리 업데이트
```
